# 카메라 관리 서브페이지 (/admin/cameras) 스펙

> 상태: 살아있는 계약 (living spec) · 독자: 스펙 작성자 / 오케스트레이터
> 역할: 관리(/admin) IA 재구성에서 "카메라 관리"를 담는 기능별 서브페이지 하나.
> 상위 설계도: [[admin-management-ia-map]] (단위 "카메라 관리 → /admin/cameras").
> 접합부(URL 체계·딥링크·뒤로가기=허브 복귀·admin 게이트)는 재정의하지 않고
> [[admin-routing-navigation-contract]]에만 의존한다.

## 목적 / 의도

관리 영역이 하나의 세로 스크롤 화면에 여러 관리 기능을 담고 있던 것을 "허브 + 기능별
서브페이지"로 재구성하면서, 이전 관리 화면의 **카메라 관리 섹션**을 독립된 딥링크 가능한
서브페이지(`/admin/cameras`)로 이관한다. 이 단위는 **순수 정보구조(IA)·표현 계층 변경**이며,
카메라 관리의 관측 가능한 동작(목록·추가·수정·삭제·상태 표시)을 **동등하게 보존**한다. 위치와
진입 경로만 바뀌고 기능 계약은 바뀌지 않는다.

상위 맵의 3가지 불변식을 **참조로 상속**한다(여기서 재정의하지 않는다):
① 행위 보존 ② 백엔드·API 무변경 ③ admin 전용 접근 경계 보존.

카메라 CRUD가 스트리밍·녹화 소비자로 전파되고 삭제가 증거를 보존하는 **행위 계약**은
[[camera-change-propagation]]이 소유한다. 이 서브페이지는 그 계약을 바꾸지 않고, 동일한
`/api/cameras` 접면을 통해 이전 섹션과 동일한 방식으로 호출한다. 카메라는 `cameras` 레지스트리
전용이며 장치(센서)·`/api/devices` 대상이 아니다([[camera-change-propagation]] 경계와 일치).

## 언어 · 런타임

브라우저 SPA(React). URL(경로)이 활성 화면의 단일 진실 소스라는 기존 라우팅 원칙을 유지한다
(접합부 계약 소유). 이 페이지는 `/admin/cameras` canonical 경로에 마운트되는 서브페이지 컴포넌트
하나다(deliverable 1:1).

## 의존 도구 · 시스템

- **카메라 API** — `/api/cameras` (기존 계약, 무변경). admin 전용.
- **접합부 계약** — 진입·복귀·admin 게이트·딥링크·미정의 경로 폴백은
  [[admin-routing-navigation-contract]]가 소유한다. 이 페이지는 그 인터페이스에만 의존한다.
- **행위 계약** — 카메라 변경 전파·삭제 증거 보존은 [[camera-change-propagation]]가 소유한다.

## 입력

- **경로 진입:** 허브에서 "카메라 관리" 항목 선택(드릴다운) 또는 `/admin/cameras` 직접 진입
  (딥링크·새로고침) — 진입 메커니즘은 접합부 계약 소유.
- **인증 컨텍스트:** admin 역할(접근 경계는 접합부 계약 소유).
- **카메라 목록:** `GET /api/cameras` 응답(각 카메라: name·location·zone·sourceType·sourceUrl·
  enabled·서버 발급 식별자).
- **폼 입력(추가/수정):** 이름, 위치, 구역, 소스 타입(rtsp | youtube), 소스 URL, (수정 한정) 활성화 여부.

## 출력 (계약)

- `/admin/cameras`에 마운트되며 admin에게만 카메라 관리 화면을 렌더한다(게이트는 접합부 계약을
  통해 상속 — 비-admin·미인증은 관리 콘텐츠를 보지 못한다).
- 이전 관리 화면의 "카메라 관리" 섹션이 제공하던 **카메라 CRUD 및 상태 표시 동작을 동등하게**
  제공한다(아래 핵심 로직·검증 단언 참조).
- 카메라 생성/수정/삭제는 이전과 동일하게 `POST/PUT/DELETE /api/cameras[/{id}]`를 호출하며,
  요청/응답 계약과 그로 인한 전파·증거 보존 행위는 변경하지 않는다.
- 허브로 돌아갈 수 있다(뒤로가기=허브 복귀 — 접합부 계약 소유).

## 핵심 로직 (동작)

이전 카메라 관리 섹션이 보장하던 관측 가능한 동작을 그대로 보존한다:

- **목록:** 등록된 카메라를 목록으로 표시한다. 로딩 중·오류·빈 목록(등록된 카메라 없음)의 세
  상태를 각각 구분해 표시한다.
- **추가:** 추가 어포던스로 입력 폼을 열어 이름·위치·구역·소스 타입·소스 URL을 입력해 카메라를
  생성한다. 이름은 필수다(누락 시 검증 오류로 생성 차단). 소스 타입은 rtsp 또는 youtube 중
  선택하며 기본값은 rtsp이고, 선택된 소스 타입에 따라 소스 URL 입력 안내(placeholder)가 바뀐다.
  취소하면 입력이 폐기되고 폼이 닫힌다.
- **수정:** 목록 항목을 편집 상태로 전환해 이름·위치·구역·소스 타입·소스 URL과 함께
  **활성화(enabled) 여부**를 변경한다. 활성화 토글은 **수정에서만** 제공된다(추가 시에는 없다).
- **삭제:** 삭제 어포던스는 곧바로 삭제하지 않고 **삭제 확인 모달**을 거친다. 확인 시에만 해당
  카메라를 삭제하고, 취소 시 아무 변경도 일어나지 않는다.
- **상태 배지:** 각 목록 항목은 소스 타입을 **대문자 배지**로 표시하고, 비활성 카메라는 별도의
  **비활성 배지**를 함께 표시한다.

'어떻게' 렌더하는지(컴포넌트 구조·상태 관리 방식)는 이 계약이 규정하지 않는다 — 위 관측 가능한
동작만 보존하면 된다.

## 검증 단언 (TDD)

- **A.** admin이 `/admin/cameras`에 진입하면 카메라 관리 화면(목록 + 추가 어포던스)이 렌더된다.
  (접근 경계·딥링크 자체는 접합부 A/B/E 스모크 참조이며 정본 판정은 접합부 소유 — 여기서는 이
  경로에서 렌더되는 콘텐츠가 카메라 관리임을 관측한다.) — 핵심
- **B.** 카메라가 하나 이상 있을 때 목록이 렌더되고, 각 항목에 수정·삭제 어포던스가 존재한다.
  목록이 비면 "등록된 카메라 없음" 상태가, 조회 실패 시 오류 상태가, 조회 중에는 로딩 상태가
  각각 구분되어 표시된다. — 핵심
- **C.** 추가 폼에서 이름을 비운 채 저장하면 검증 오류로 생성이 차단되고, `POST /api/cameras`가
  발생하지 않는다. 이름을 포함한 유효 입력으로 저장하면 name·location·zone·sourceType·sourceUrl을
  담아 `POST /api/cameras`가 호출된다. — 핵심
- **D.** 추가 폼의 소스 타입 선택지는 rtsp·youtube 두 가지이고 기본값은 rtsp이며, 선택된 타입에
  따라 소스 URL 입력 안내(placeholder)가 rtsp용/youtube용으로 전환된다. — 일반
- **E.** 수정 상태에서는 **활성화(enabled) 토글**이 제공되어 그 값을 포함해 `PUT /api/cameras/{id}`가
  호출된다. 추가 폼에는 활성화 토글이 없다. — 핵심
- **F.** 삭제 어포던스는 즉시 삭제하지 않고 삭제 확인 모달을 띄운다. 모달에서 확인해야
  `DELETE /api/cameras/{id}`가 호출되고(성공 시 204), 취소하면 어떤 삭제 요청도 발생하지 않는다.
  삭제 요청이 실패하면 오류를 표면화하고 성공으로 오인하지 않는다(#103 무음 삼킴 방지). — 핵심
- **G.** 각 목록 항목은 소스 타입을 대문자 배지로 표시하고, `enabled=false`인 카메라에는 비활성
  배지가 추가로 표시된다. — 일반
- **H.** 이 페이지가 호출하는 `/api/cameras` 요청/응답 계약(경로·메서드·요청 필드 집합)은 재구성
  이전 카메라 관리 섹션과 동일하다(백엔드·API 무변경 불변식; 각 호출은 동일 admin 인증 헤더를
  부착한다). — 핵심
