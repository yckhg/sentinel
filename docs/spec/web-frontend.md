# web-frontend 스펙

> 상태: 살아있는 계약 (living spec) · 독자: 스펙 작성자

## 목적 / 의도

위험 작업장의 관리자·작업자가 **모바일 브라우저 하나로** 현장 안전 상태를 실시간 파악하고 즉시 대응할 수 있게 한다. 구체적으로:

- 위기(crisis) 이벤트가 발생하면 열려 있는 화면에 **사람이 놓칠 수 없는 형태로 즉시** 표시된다.
- CCTV 라이브 영상과 최근 녹화를 브라우저에서 바로 재생할 수 있다.
- 사고 이력 확인, 장비 재시작, 119 신고 등 대응 행동을 화면에서 수행할 수 있다.
- 계정 없는 외부인에게도 임시 링크 하나로 라이브 영상만 제한 공개할 수 있다.

## 언어 · 런타임

- TypeScript + React 18 SPA. Vite로 정적 번들을 빌드한다.
- 런타임은 **사용자의 모바일/데스크톱 브라우저**다. 서버 측 렌더링 없음.
- 배포 형태는 nginx 컨테이너 1개: 정적 파일 서빙 + 백엔드/스트리밍 리버스 프록시.
- HLS 재생: MSE 지원 브라우저는 hls.js, Safari 계열은 브라우저 내장 HLS 재생기를 사용한다. 두 경로 모두 보장된다.

## 의존 도구 · 시스템

이 서비스는 아래 접면을 **소비**만 하며, 접면 계약의 소유자는 각 interface 문서다. 이 스펙에서 재정의하지 않는다.

| 접면 | 소유 문서 |
|------|-----------|
| REST API (`/api/*`, `/auth/*`) + WebSocket (`/ws`) | `docs/spec/interface-web-api.md` |
| HLS 스트리밍 (`/live/*`) | `docs/spec/interface-streaming.md` |

기타 의존:
- 브라우저 Geolocation API (119 신고 시 현재 좌표 표시, 실패해도 신고 흐름은 막지 않음)
- 브라우저 `tel:` URL 스킴 (119 발신)
- `localStorage` (JWT 보관)

## 입력

- **사용자 입력**: 로그인/가입 폼, 탭 전환, 카메라 타일 탭(확대), 타임라인 조작, 사고 확인/조치 입력, 장비 재시작 확인, 119 신고 확인, 관리 CRUD 폼.
- **URL**: `/view/{token}` (뷰어 임시 링크), `/register` (+ `?invite=` 초대 토큰), `/login` (로그인), 메인 앱의 각 탭은 고유 경로(예: CCTV/사고이력/관리/설정)를 가진다. 정의되지 않은 경로는 404 화면으로 처리한다.
- **web-backend 응답**: 카메라 목록, 사고 이력, health/devices 상태, 설정값, 녹화 구간/보관 목록, **미해제(open+acknowledged) 위기 목록**(재접속 백필용, `crisis_alert` payload 동형) (shape는 interface-web-api.md 소유).
- **WebSocket 푸시**: `crisis_alert` 타입 메시지 (shape는 interface-web-api.md 소유).
- **HLS 세그먼트**: `/live/*` 및 녹화 재생 URL (규격은 interface-streaming.md 소유).

## 출력 (계약)

사용자에게 보장되는 것:

1. **모바일 뷰포트 지원** — 페이지는 `width=device-width` 뷰포트로 렌더되며, 모든 기능(라이브 뷰, 사고 처리, 관리)이 모바일 화면에서 조작 가능하다. UI 언어는 한국어다.

2. **위기 알림 즉시 표시** — WebSocket 연결이 살아 있는 동안, `crisis_alert` 메시지 수신 즉시(추가 네트워크 왕복 없이) 화면 상단에 전폭 배너가 나타난다. 배너는 발생 시각(KST)·설명을 담고, 사용자가 직접 닫기 전까지 유지되며, 다중 위기는 누적 표시된다(최신이 위). 배너에는 119 신고 버튼이 함께 붙는다. 이 배너는 로그인 후 어떤 탭에 있어도 표시된다.
   - **배너 식별 키**: 각 배너의 동일성 판정 키는 위기의 `incidentId`다. 라이브 수신(`crisis_alert`)과 재접속 백필(출력 3-(c)) 양 경로 모두 이 키로 중복을 판정하므로, 같은 `incidentId`의 위기는 어느 경로로 들어와도 배너가 하나만 존재한다.

3. **위기 알림 채널 상태 가시성·자기복구·백필** — 안전 크리티컬 채널이므로 연결 상태와 복구를 사용자가 항상 인지할 수 있어야 한다.
   - (a) **상태 가시성 (소켓 상태만 반영)**: WebSocket 연결 상태(연결됨 / 끊김 / 재접속 중)는 로그인 후 어떤 탭에 있어도 화면에 항상 시각적으로 표시된다. 이 표시는 **소켓의 연결 상태만** 반영한다. 즉, 재접속에 성공해 소켓이 열리는 즉시 "연결됨"으로 돌아가며, 뒤따르는 백필((c))의 성공·실패·진행 여부와는 무관하게 결합되지 않는다. 끊김·재접속 중 상태는 사람이 놓칠 수 없는 형태(색·문구)로 나타나며, 이 구간 동안 위기 알림이 무음으로 유실될 수 있음을 사용자가 인지할 수 있다. 이 상태는 관측 가능한 DOM 마커 `data-ws-state`(`connected`/`disconnected`/`reconnecting`)로도 노출된다(단언 O).
   - (b) **자기복구**: WS가 끊기면 지수 백오프(1s → 2s → 4s → 최대 8s)로 자동 재접속한다. 즉, 네트워크 순단 후 알림 채널은 최대 수 초 내에 복구된다.
   - (c) **재접속 백필 (best-effort, 상태 표시와 분리)**: 재접속에 성공하면, 끊긴 구간 동안 놓쳤을 수 있는 미해제(=`resolved`가 아닌) 위기를 백엔드에서 재조회(backfill)하여 배너 상태와 재동기화한다. 백필은 소켓 상태 표시((a))와 분리된 **best-effort 동작**이다.
     - **조회 범위**: 백필은 `status ∈ {open, acknowledged}`(즉 아직 `resolved`되지 않은) 위기만 조회한다. `acknowledged`도 아직 해제 전이므로 미해제로서 배너가 유지되어야 한다.
     - **데이터 원천·사상 규칙**: 백필의 데이터 원천은 `docs/spec/interface-web-api.md`가 소유하는 **미해제 위기 조회 접면**이며, 이 접면은 `crisis_alert` 배너 payload와 **동형**(`incidentId` + 중첩 `site` 객체 포함)으로 응답함을 계약한다. 프론트는 백필 항목을 이 동형 구조 그대로 배너 payload로 사상한다 — 따라서 배너의 사업장 주소·좌표 등 파생 표시(예: 119 다이얼로그의 주소 fallback, 출력 10)가 라이브 경로와 동일하게 채워지며 깨지지 않는다.
     - **재동기화(진짜 sync, additive 아님)**: 백필 결과는 현재 배너 집합과 `incidentId` 기준으로 양방향 정합된다. (1) 백필에 있는데 아직 배너가 없는 미해제 위기는 배너로 **추가**한다. (2) 이미 표시 중인 동일 `incidentId` 위기는 **중복 배너를 만들지 않는다**. (3) 현재 배너 중 백필 결과에 **더 이상 존재하지 않는(=끊긴 구간에 `resolved`된)** 위기의 배너는 **제거**한다 — 스테일 배너가 영구 잔존하지 않는다.
     - **실패 경로**: 백필 조회가 실패하면(네트워크·타임아웃·5xx) 소켓 상태 표시는 (a)에 따라 이미 "연결됨"이며, 배너 집합은 실패 직전 상태를 그대로 유지한다(임의 소거 금지). 실패 시 다음 재접속 이벤트에서 다시 시도하며, 실패가 위기 배너 수신(라이브 경로)이나 다른 탭 기능을 차단하지 않는다.

4. **CCTV 라이브 재생** — 로그인 사용자는 CCTV 탭에서 등록된 전 카메라를 그리드로 보고, 각 타일에서 라이브 HLS가 자동 재생(음소거·인라인)된다. 라이브는 저지연 설정으로 재생된다. 카메라가 `disconnected`이거나 재생이 치명적으로 실패하면 해당 타일에 "연결 끊김"이 표시되고 재생을 시도하지 않는다. 카메라 목록·상태는 30초 주기로 자동 갱신된다.

5. **녹화 재생·보관** — 카메라를 확대하면 최근 1시간 타임라인이 표시된다. 타임라인의 시점을 탭하면 그 시점부터(최대 5분 블록) 녹화가 같은 타일에서 재생되고 "녹화 재생" 표식이 붙으며, 언제든 라이브로 복귀할 수 있다. 구간을 선택해 보관(archive)을 요청할 수 있고, 완료된 보관본은 mp4로 다운로드된다.

6. **인증 게이트·딥링크 복귀** — 유효한 JWT 없이 메인 앱의 보호 경로(canonical 탭 경로: `/cctv`·`/incidents`·`/admin`·`/settings`)에 접근하면 canonical `/login` 경로로 리다이렉트되어 로그인 화면이 표시된다(URL이 보호 경로 그대로 남지 않는다). 이때 원래 요청 경로가 `returnTo`로 보존되어, 로그인 성공 후 그 경로(유효한 앱 내 canonical 경로일 때만)로 복귀한다 — 없거나 무효면 기본 탭 `/cctv`로 간다. 보호→`/login` 및 로그인→복귀 전이는 history replace로 처리되어 뒤로가기 루프가 생기지 않는다. 저장된 토큰이 만료 상태면 앱 진입 시 폐기되고 `/login`으로 돌아간다. 관리자 전용 조작(사고 확인/조치 등)은 admin 역할 토큰에서만 노출된다.

7. **뷰어 임시 링크** — `/view/{token}` 접속 시 토큰이 곧 인증이다. 토큰이 유효하면 라이브 그리드만 제공되고 네비게이션·관리·설정 UI는 일절 노출되지 않는다. 토큰이 무효/만료면 "접근 불가" 화면 외에 아무것도 보이지 않는다.

8. **사고 이력** — 사고 목록은 최신순·20건 단위 페이지네이션으로 표시되고, 기간·상태(미확인/확인됨/조치완료)로 필터할 수 있다. 각 사고 카드는 상태 배지·테스트 여부를 표시하며, 조치완료 건은 조치 내용과 해제 주체(웹 해제 / 센서 버튼 해제)를 구분해 보여준다. 조치 완료 처리에는 조치 내용 입력이 강제된다.

9. **장비 재시작 안전장치** — 재시작 명령은 2단계 확인(확인 → 사유 입력 후 최종 확인)을 모두 통과해야만 전송된다. 카메라에 장비 매핑 정보가 없으면 확인 다이얼로그 자체가 열리지 않고 안내만 표시된다(잘못된 대상으로 명령이 나가지 않음).

10. **119 신고** — 119 버튼은 확인 다이얼로그를 거친 뒤에만 전화 발신을 시작한다. 다이얼로그에는 현재 위치(GPS 좌표, 실패 시 사업장 주소, 그것도 없으면 "주소 정보 없음")가 표시된다. 위치 획득 실패가 신고를 막지 않는다.

11. **운영 상태 가시성** — 관리 탭에서 서비스/센서 통합 health가 15초 주기, 장비 목록이 10초 주기로 자동 갱신된다. 장비 목록의 생존(alive) 표시는 마지막 신호로부터의 경과가 임계 미만인지로 프론트가 자체 판정하며, 이 임계는 백엔드 health 설정값 `health.sensor_alive_threshold_sec`(`GET /api/settings`, 기본 60초)를 런타임에 읽어 초→ms로 환산해 사용한다. 조회 실패 시(예: 비관리자에게는 `GET /api/settings`가 `403`) 60초 기본값으로 폴백한다. 이 폴백 기본값이 백엔드 기본값(60초)과 동일하므로, 비관리자 화면과 health API의 센서 생존 판정은 일치한다. 관리자가 설정값을 변경하면 관리자 화면의 생존 판정이 새 값을 반영한다.

12. **응답성 보장** — 모든 API 호출에는 타임아웃(기본 10초, 로그인 30초)이 걸려 있어, 백엔드 무응답 시 무한 로딩 대신 한국어 안내 메시지가 표시된다. 목록 화면은 로딩/오류/빈 상태를 각각 구분해 표시한다.

13. **단일 진입점 서빙** — 컨테이너는 80 포트 하나로 정적 SPA, `/api/`·`/auth/`·`/ws`(백엔드), `/live/`(스트리밍) 프록시를 모두 제공한다. 프론트 코드는 절대 URL을 쓰지 않으므로 어떤 호스트/포트로 노출돼도 동작한다. `/healthz`는 항상 200과 서비스 식별 JSON을 반환한다.

## 핵심 로직 (동작)

- **라우팅**: URL 경로가 현재 뷰의 정본(single source of truth)이다. `/view/{token}` → 뷰어, `/register` → 가입, `/login` → 로그인. 메인 앱의 4개 탭은 각각 아래 **canonical 경로 문자열**을 가진다: CCTV=`/cctv`, 사고이력=`/incidents`, 관리=`/admin`, 설정=`/settings`. 이 문자열이 딥링크·복원·리다이렉트 판정의 기준이다. 탭 전환은 브라우저 히스토리에 반영된다. 새로고침·뒤로가기·딥링크 시 URL이 가리키는 뷰가 그대로 복원된다(특정 탭으로 리셋되지 않는다). 위 어느 경로에도 매칭되지 않는 경로는 404 화면을 표시한다(메인 앱으로 흡수하지 않는다).
- **미인증 딥링크·복귀(returnTo)**: 미인증 상태로 보호 경로(canonical 탭 경로)에 접근하면 `/login`으로 리다이렉트하되, 원래 요청 경로를 `returnTo`로 보존한다. 로그인 성공 시 보존된 `returnTo`가 유효한 canonical 보호 경로면 그 경로로 복귀하고, 없거나 무효(오픈 리다이렉트 방지: 앱 내 canonical 경로만 허용)면 기본 탭(`/cctv`)으로 간다. 보호→`/login` 리다이렉트와 로그인 성공→`returnTo` 복귀는 모두 **history replace** 시맨틱으로 수행한다(뒤로가기 시 `/login`이나 리다이렉트 중간 항목으로 되돌아가는 루프가 생기지 않는다).
- **위기 배너**: 앱 루트에 상주하는 단일 WebSocket 연결이 `crisis_alert` 메시지를 받아, `incidentId` 기준으로 아직 배너가 없으면 목록 맨 앞(최신 위)에 추가한다(동일 `incidentId`면 중복 추가하지 않음). 파싱 불가 메시지는 무시한다. 배너 제거 경로는 두 가지다: (1) 사용자 닫기, (2) 재접속 백필 재동기화에서 해당 위기가 미해제 목록에 더 이상 없음이 확인될 때(=끊긴 구간에 `resolved`됨). 즉 스테일(해제됨) 배너는 백필 시 자동 제거된다.
- **HLS 재생기**: 재생 URL이 바뀌면 기존 재생 세션을 완전히 파기하고 새로 연다. 라이브는 저지연 모드, 녹화 재생은 일반 모드. 자동재생 차단은 조용히 허용(사용자 제스처로 재생 가능).
- **확대/재생 상태**: 확대는 한 번에 한 카메라. 확대 해제 또는 다른 카메라 확대 시 녹화 재생 상태는 라이브로 초기화된다.
- **보관 진행 추적**: 보관 요청 후 3초 간격 폴링으로 완료/실패를 반영하고, 5분 후 폴링을 안전 중단한다.
- **토큰 취급**: JWT는 localStorage에 보관하고 모든 인증 요청에 Bearer로 첨부한다. 만료 판정은 클라이언트에서 exp 클레임으로만 한다(서명 검증 없음 — 접근 제어의 실체는 백엔드). admin 여부도 토큰 클레임에서 읽어 UI 노출만 제어한다.
- **영속 상태 없음**: 서버 상태의 캐시를 두지 않는다. 모든 화면은 진입/주기 폴링 시 백엔드에서 다시 읽는다.

## 검증 단언 (TDD)

각 단언은 OK/NOK 판정 가능한 관측 명제다. "주입"은 백엔드 stub 또는 실서버로 수행한다.

- **A. 뷰포트** — 배포된 `index.html` 응답에 `<meta name="viewport" content="width=device-width, ...">`가 존재한다.
- **B. 인증 게이트** — localStorage에 토큰이 없거나 exp가 과거인 토큰만 있는 상태에서 `/` 접속 → 로그인 화면이 렌더되고, 만료 토큰은 localStorage에서 제거되어 있다.
- **C. 위기 배너 즉시성·키 dedup** — 로그인 상태에서 WS로 `{"type":"crisis_alert","payload":{...}}` 1건 주입 → 추가 HTTP 요청 없이 상단 배너가 나타나고, 설명·KST 시각·119 버튼·닫기 버튼이 포함된다. 서로 다른 `incidentId`의 2건째 주입 → 배너 2개 누적(최신 위). 닫기 클릭 전까지 탭을 전환해도 배너가 유지된다. 동일 `incidentId`를 2번 주입 → 배너는 1개만 유지된다(라이브 경로 dedup 키 = `incidentId`).
- **D. WS 자기복구** — WS 연결을 강제 절단 → 재접속 시도가 1s, 2s, 4s, 8s 간격으로 발생하고 이후 8s를 넘지 않는다. 재접속 성공 후 주입한 `crisis_alert`가 다시 배너로 표시된다.
- **E. 라이브 렌더** — `GET /api/cameras`가 N개(≥1) 카메라를 반환하도록 주입 → 카메라 타일 N개가 렌더되고, `status:"connected"` 타일은 HLS 로드를 시작하며, `status:"disconnected"` 타일은 "연결 끊김"을 표시하고 스트림 요청을 보내지 않는다.
- **F. 목록 자동 갱신** — CCTV 탭을 열어둔 채 관측 → `/api/cameras` 요청이 30±5초 간격으로 반복된다.
- **G. 타임아웃 UX** — `/api/cameras` 응답을 10초 이상 지연시키면 → 무한 로딩이 아니라 "서버 응답 시간이 초과되었습니다..." 안내가 표시된다.
- **H. 뷰어 링크** — 유효 토큰으로 `/view/{token}` 접속 → 라이브 그리드만 표시되고 탭바·관리 UI가 DOM에 없다. `verify`가 401을 반환하는 토큰으로 접속 → "링크가 만료되었거나 유효하지 않습니다"가 표시되고 카메라 요청이 발생하지 않는다.
- **I. 재시작 안전장치** — (1) siteId/deviceId가 없는 카메라에서 재시작 버튼 클릭 → 안내 alert만 뜨고 `/api/equipment/restart` 요청이 발생하지 않는다. (2) 매핑이 있는 카메라에서 1단계 확인만 하고 취소 → 요청 없음. (3) 2단계 최종 확인까지 완료 → 해당 카메라의 siteId/deviceId가 담긴 POST 1건이 발생한다.
- **J. 사고 이력 계약** — `GET /api/incidents`에 `{data:[...], pagination:{total:57,...}}` 주입 → "총 57건"과 20개 카드가 렌더되고, "더 보기" 클릭 시 `page=2` 요청이 발생한다. `status:"resolved"` + `resolvedByKind:"sensor_button"` 건은 "센서 버튼" 해제 표시를 포함한다.
- **K. admin 게이트** — role이 admin이 아닌 토큰으로 사고 목록 조회 → 확인/조치 버튼이 렌더되지 않는다. admin 토큰 → open 건에 "확인", 미해결 건에 "조치 완료" 버튼이 렌더되고, 조치 내용이 비어 있으면 resolve 요청이 전송되지 않는다.
- **L. 119 게이트** — 119 버튼 클릭 → `tel:` 발신 없이 확인 다이얼로그가 먼저 뜬다. Geolocation을 거부해도 다이얼로그는 정상 동작하며(주소 fallback), "119 전화 걸기" 클릭 시에만 `tel:119` 발신이 시작된다.
- **M. 운영 폴링** — 관리 탭을 열어둔 채 관측 → `/api/health`가 15±3초, `/api/devices`(또는 `/devices/all`)가 10±3초 간격으로 반복 호출된다.
- **N. 서빙 계약** — 컨테이너 80 포트에서 (1) `GET /healthz` → 200 + `{"status":"ok",...}`, (2) 임의 SPA 경로(`GET /view/abc`) → `index.html` 반환, (3) `/api/`·`/auth/`·`/ws`·`/live/` 요청이 각각 백엔드/스트리밍으로 프록시된다.
- **O. 연결 상태 가시성** — 상태 표시 요소는 관측 가능한 DOM 마커 `data-ws-state`를 노출하며 값은 `connected` / `disconnected` / `reconnecting` 중 하나다(색·문구는 이 상태에 종속). 로그인 상태에서 관측 → 마커가 존재하고 `data-ws-state="connected"`다. WS를 강제 절단 → `data-ws-state`가 `disconnected` 또는 `reconnecting`으로 바뀌고 색·문구가 함께 변한다. 재접속 성공(소켓 open) 즉시 → 백필의 성공·실패와 무관하게 `data-ws-state="connected"`로 돌아간다. (이 마커는 어떤 탭에 있어도 DOM에 유지된다.)
- **P. 재접속 백필 재동기화** — 미해제 위기 조회 접면(interface-web-api.md 소유, `crisis_alert` payload 동형: `incidentId` + 중첩 `site`)을 stub로 둔다.
  - (1) **추가**: WS 절단 상태에서 접면이 `status:"open"` 위기 1건을 반환하도록 주입 → 재접속 성공 직후 프론트가 조회하고 그 위기가 배너로 표시되며, 배너의 주소 fallback(출력 10) 등 파생 표시가 라이브 경로와 동일하게 채워진다.
  - (2) **acknowledged 포함**: 접면이 `status:"acknowledged"` 위기를 반환 → 해당 위기도 미해제로서 배너에 포함된다(open만이 아님).
  - (3) **중복 없음**: 이미 동일 `incidentId` 배너가 표시 중이면 백필로 중복 배너가 추가되지 않는다(dedup 키 = `incidentId`).
  - (4) **스테일 제거(진짜 sync)**: 배너로 표시 중이던 `incidentId`가 백필 응답에 더 이상 없으면(끊긴 구간에 `resolved`됨) 해당 배너가 제거된다.
  - (5) **표시기 분리**: 백필 접면이 오류(5xx/타임아웃)를 반환해도 연결 상태 표시는 재접속 성공 즉시 "연결됨"이며, 기존 배너 집합은 임의로 소거되지 않고 유지된다.
- **Q. URL 라우팅·딥링크** — (1) CCTV가 아닌 탭(예: `/admin`)으로 이동하면 URL이 그 canonical 경로로 바뀌고, 새로고침 시 관리 탭이 그대로 복원된다(CCTV로 리셋되지 않는다). (2) 탭 이동 후 브라우저 뒤로가기 → 직전 탭으로 돌아간다. (3) 미인증 상태로 보호 경로에 접속 → `/login`으로 리다이렉트되고 URL이 보호 경로로 남지 않는다. (4) 어느 경로에도 매칭되지 않는 경로(예: `GET /nonexistent`)로 앱 내 이동 → 404 화면이 표시되고 메인 앱/CCTV로 흡수되지 않는다. (5) **canonical 경로**: 각 탭의 URL 문자열은 정확히 `/cctv`·`/incidents`·`/admin`·`/settings`다. (6) **returnTo 복귀**: 미인증으로 `/admin` 딥링크 → `/login` 후 로그인 성공 → `/admin`으로 복귀한다. `returnTo`가 없거나 앱 밖/무효 경로면 `/cctv`로 간다. (7) **history replace(루프 방지)**: 미인증 `/admin` 딥링크→`/login`→로그인→`/admin` 복귀 후 브라우저 뒤로가기 → `/login`이나 리다이렉트 중간 항목으로 돌아가지 않는다(무한 루프 없음).
- **R. 아카이브 status 소비자 판정** — 아카이브 목록 UI(RecordingTimeline)가 `GET /api/archives` 응답의 `status` enum을 소비하는 방식을 관측한다(값 정의 SSOT=recording.md, 소비자 의무=interface-web-api.md §계약8 "아카이브 status 소비자 계약"). (1) **6종 전부 구별**: enum 6종 `{protecting, pending, finalizing, processing, completed, failed}`을 각각 담은 항목을 주입 → 각 항목이 서로 구별되는 상태 라벨/배지(예: `data-status` 마커 또는 상태 문구)로 렌더되며, 어느 하나도 다른 상태로 조용히 오인되지 않는다. (2) **미지 상태 안전 fallback**: enum 밖의 미지 `status`(예: `"unknown"`)를 담은 항목을 주입 → 해당 항목은 미완료(진행 중)로 표시되고, 완료(`completed`)로 표시되지 않으며 다운로드 어피어던스(다운로드 버튼·링크)가 그 항목의 DOM에 존재하지 않는다. (3) **`failed` 오류 종단**: `status:"failed"` 항목을 주입 → 실패임을 나타내는 라벨과 사유가 오류 종단으로 사용자에게 노출되고, 이 항목 역시 다운로드 어피어던스를 노출하지 않는다.

## ⚠️ 리뷰 필요 (의도 불확실)

1. **~~녹화 타임라인의 사고 마커가 표시되지 않을 가능성~~ (해소됨)** — 타임라인이 `GET /api/incidents` 응답에서 계약상 배열 필드인 `data`(`{data, pagination}` 엔벌로프)를 읽도록 정렬되어, 사고 마커 배열이 정상적으로 채워진다.
2. **세션 중 401에 대한 로그아웃/리다이렉트 부재** — 서비스 가이드에는 "요청 실패 시 login 리다이렉트"가 명시돼 있으나, 코드상 401 응답을 받아도 토큰 폐기·로그인 화면 전환이 일어나지 않고 "HTTP 401" 오류 문구만 표시된다. 만료 판정은 앱 최초 진입 시 1회뿐. 의도된 축소인지 미구현인지 확인 필요.
3. **`incident_resolved` WS 이벤트 라이브 미소비 (부분 해소)** — 재접속 백필이 미해제 목록에 없는(=`resolved`된) 배너를 재동기화 시 제거하도록 계약화되어(출력 3-(c)·핵심로직 위기배너·단언 P-(4)), 재접속을 거치면 스테일 배너가 사라진다. 다만 **소켓이 계속 연결된 채** 사고가 해제되는 경우, `docs/spec/interface-web-api.md`가 정의한 `incident_resolved` 브로드캐스트를 프론트가 라이브로 소비하지 않으면, 다음 재접속 전까지는 배너가 즉시 사라지지 않는다. ⚠️ 라이브 `incident_resolved` 소비로 배너를 즉시 제거하는 것까지 계약할지(안전상 권장) 확인 필요 — 이 경우 dedup·제거 키도 `incidentId`로 백필 경로와 동일하게 맞춘다.
4. **~~WS 재접속 시 놓친 위기 이벤트 미복원~~ (해소됨)** — 계약이 개정되어, 재접속 성공 시 미해제(`status ∈ {open, acknowledged}`) 위기를 백엔드에서 재조회(backfill)해 배너와 **양방향 재동기화**(추가+스테일 제거)하도록 규정한다(출력 계약 3-(c), 검증단언 P). 배너 dedup·제거 키는 라이브·백필 공통으로 `incidentId`다(출력 계약 2, 단언 C·P). 연결 끊김 구간은 상태 표시(DOM 마커 `data-ws-state`)로 항상 노출되고, 이 표시는 소켓 상태만 반영하며 백필과 분리된다(출력 계약 3-(a), 검증단언 O). 백필의 데이터 원천은 `docs/spec/interface-web-api.md`가 신설·소유하는 **미해제 위기 조회 접면**이며 `crisis_alert` payload 동형(`incidentId` + 중첩 `site`)으로 응답하도록 계약한다 — 이 접면 계약의 확정은 interface 문서 소관이다.
5. **녹화 타임라인 창이 열림 시점에 고정** — "최근 1시간" 창이 타임라인을 연 순간으로 고정되어 갱신되지 않는다. 오래 열어두면 현재 시각과 어긋난다. 의도(스냅샷) 여부 확인 필요.
6. **로그인 상태에서 `/register` 접속 시 로그인/가입 화면 강제** — 유효 토큰이 있어도 `/register`에서는 메인 앱 대신 가입 화면이 렌더된다. 초대 수락 흐름을 위한 의도로 보이나 명시 확인 필요.
