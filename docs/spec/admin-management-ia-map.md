# 관리(/admin) IA 재구성 스펙 설계 맵

> 상태: 살아있는 계약 (living spec) · 독자: 스펙 작성자 / 오케스트레이터
> 역할: 관리 영역을 "허브 + 기능별 서브페이지"로 분할한 최상위 설계도.
> 이 문서는 흐름(무엇이 어떤 순서로 무엇을 보장하는가)과 단위 경계만 고정한다.
> 각 단위의 내부 구현과 분해 중 파생되는 하위 스펙은 구현 주체(오케스트레이터)에 위임한다.
> 인간 검토면 = 이 맵의 흐름 + 각 단위 in/out·단언 + 의도적 검증 스킵 선언. 그 외는 위임.

## 목적 / 의도

관리 영역이 성격이 다른 관리 기능 다수를 **하나의 세로 스크롤 화면**에 담아, 특정 기능을
찾으려면 과도한 스크롤이 필요하고 발견성이 나쁘다. 이를 **관리 허브(기능 목록) → 기능별
서브페이지(드릴다운)** 로 재구성해, "한 화면 = 한 관심사" 원칙과 딥링크 가능한 URL 체계를
확보한다.

이 재구성은 **순수 정보구조(IA)·표현 계층 변경**이다. 아래 3가지가 이 설계 맵 전체를 관통하는
불변식이며, 모든 단위 스펙이 상속한다:

1. **행위 보존 (behavior-preserving):** 각 서브페이지는 재구성 이전 대응 기능의 **관측 가능한
   동작(CRUD·검증·목록·상태 표시·발송 등)을 동등하게** 보장한다. 재구성은 위치·내비게이션만
   바꾸고 기능 계약을 바꾸지 않는다.
2. **백엔드·API 무변경:** 어떤 서버 엔드포인트·요청/응답 계약도 바뀌지 않는다. 각 페이지는
   기존과 동일한 API를 동일한 방식으로 호출한다.
3. **접근 경계 보존:** 관리 영역 전체(허브 + 모든 서브페이지)는 기존과 동일한 **admin 전용**
   게이트를 유지한다. 비-admin은 허브·서브페이지 어디에도 도달하지 못한다.

## 분할 원칙

- 한 단위(서브페이지)의 in/out은 다른 단위의 내부를 몰라도 성립한다. 단위들이 공통으로 기대는
  것(URL 체계·드릴다운·뒤로가기·admin 게이트)은 **[[admin-routing-navigation-contract]]**
  (라우팅·내비게이션 접합부 계약)이 별도로 소유한다.
- 각 작업 단위 스펙의 검증 단언 = 그 단위 작업 지시의 인수 게이트 = spec-tdd의 OK/NOK 게이트.
- 단위 경계는 **산출물(deliverable)** 을 따른다 — 서브페이지 컴포넌트 하나 = 스펙 하나(1:1).
- "딱 붙는 관심사"만 한 페이지로 합친다: 계정+초대 → 사용자(동일 수명주기), 현장정보+시스템설정
  → 시스템(**둘 다 소형 admin 설정면**이라 한 화면에 합침). 그 외는 1관심사 1페이지.
  - 반대로 **비상 신호 시뮬레이션(test-alert)** 과 **알림 채널 테스트 발송(notify-test)** 은 분리
    유지한다 — 전자는 전체 알림 체인을 팬아웃 발동하고 후자는 단일 채널·단일 대상 단건 발송으로
    **팬아웃 유무가 상반**되므로 관심사가 붙지 않는다.
- **공유 산출물 소유권(중복 방지):** 여러 리프에 걸치는 공유 코드는 리프가 각자 복제하지 않고
  마스터가 소유한다. (a) `pages/ManagementPage.tsx`는 **동결된 행위 참조**로 남고 어느 리프도
  편집하지 않는다 — 각 리프는 `pages/admin/<Slug>Page.tsx` **신규 파일**을 작성한다. 마스터가
  App.tsx `case 'management'`를 허브+slug 해석기로 교체하며 ManagementPage 렌더 경로를 제거하고,
  기존 `ManagementPage.*.test.tsx`의 은퇴도 마스터가 소유한다. (b) contacts·system 두 리프가
  공유하는 전화번호 검증 규칙(`^01[016789]-\d{3,4}-\d{4}$`)은 `utils/phone.ts` 공유 유틸로 두며
  **소유자=마스터**, 두 리프는 이를 참조한다.
- **구현 순서(선후):** 접합부+허브(마스터 — App.tsx 라우트 테이블·게이트·slug 해석기)를 1순위로
  구현·머지하고, 그 후 리프를 병렬 구현한다. App.tsx는 모든 리프 라우팅 단언의 공유 파일이므로,
  마스터가 들어오기 전에는 `/admin/<slug>` 딥링크가 404가 되어 리프의 라우팅계열 단언(A/B/C·
  게이트)이 자기 워크트리에서 GREEN이 될 수 없다. 따라서 그 단언들은 **통합 시점 게이트로만
  판정**한다.
- **워크트리 배치 힌트:** health는 유일하게 load-bearing 개선(#112 다열 배치)과 전용 픽스처를
  가지므로, 다른 리프와 섞이지 않게 **독립 워크트리에 배치하고 가장 늦게** 통합하는 것을 권장한다.

## 단위 트리

```
관리(/admin) IA
├─ [접합부] 라우팅·내비게이션 계약  (횡단 — 모든 단위가 의존)
│     URL 체계 /admin, /admin/<slug> · 딥링크 · 뒤로가기=허브 복귀 · admin 게이트 · open-redirect 가드 확장
├─ 관리 허브 (menu)                → /admin
│   그룹 라벨 + 전체 기능 목록, 각 항목 탭 시 해당 서브페이지로 이동
├─ 장치
│   ├─ 장비(센서) 관리             → /admin/devices
│   ├─ 카메라 관리                 → /admin/cameras
│   └─ 장비 상태·예외             → /admin/health      (#112 예외 목록 레이아웃 여기로 흡수)
├─ 알림·연락
│   ├─ 비상연락망                  → /admin/contacts
│   ├─ 비상 신호 시뮬레이션        → /admin/test-alert
│   └─ 알림 채널 테스트 발송       → /admin/notify-test
├─ 시스템
│   ├─ 시스템 설정 (현장+시스템)   → /admin/system
│   └─ 사용자 (계정+초대)          → /admin/users
└─ 저장·CCTV
    ├─ 저장소 관리                 → /admin/storage
    └─ 임시 CCTV 링크              → /admin/cctv-links
```

## 각 단위 — 한 줄 책임 · in/out · 상태

전 단위 공통 출력 계약(상속): 자신의 canonical path에 마운트되고, admin에게만 렌더되며, 허브로
돌아갈 수 있고, 대응 이전 기능과 행위가 동등하다(위 불변식 1·3 + 접합부 계약).

공통 참고:
- **`/api/settings` 공유 읽기 접면:** devices·system 두 리프가 같은 `/api/settings`를 읽되 서로
  다른 키를 읽는다(devices=`health.sensor_alive_threshold_sec`, system=`site_url`). 쓰기·응답
  shape는 바뀌지 않는다.
- **독립 로딩/오류:** 각 서브페이지는 자신의 로딩/오류 상태를 독립적으로 가진다. 재구성 이전
  페이지-레벨 로딩 결박은 **의도적으로 미승계**한다(발견성·독립성 개선).

### 관리 허브 — 전체 기능 목록을 그룹별로 보여주고 각 항목을 서브페이지로 연결
- 입력: 없음(admin 인증). 
- 출력(계약): 4그룹([장치] [알림·연락] [시스템] [저장·CCTV]) 라벨 아래 10개 항목을 모두 나열; 각 항목
  탭 → 해당 canonical path로 내비게이트. 어떤 기능도 목록에서 누락되지 않는다.
- 상태: 작업 단위 스펙 위임 후보

### 장비(센서) 관리 — /admin/devices
- 입력/의존: `/api/devices` (목록·등록·삭제 등 기존 계약).
- 출력(계약): 이전 "장비(센서) 관리" 섹션의 센서 장치 CRUD·sticky 삭제·재활성 동작을 동등 보존.
- 상태: 위임 후보

### 카메라 관리 — /admin/cameras
- 입력/의존: `/api/cameras`.
- 출력(계약): 이전 카메라 관리 섹션의 카메라 CRUD 동작 동등 보존.
- 상태: 위임 후보

### 장비 상태·예외 — /admin/health
- 입력/의존: `/api/health/summary`, `/api/health/events`, `/api/devices`, `/api/cameras`.
- 출력(계약): 이전 건강/예외 패널의 예외(이상·오프라인) 장비 목록·상태 전이 이력·카메라 연결
  상태를 동등 보존. **#112(예외 목록이 다수일 때 단일 컬럼 나열 불편)의 레이아웃 개선은 이
  페이지 단위 안에서 다룬다** — 단, 이 맵의 행위보존 불변식과 별개의 개선이므로 해당 페이지
  스펙이 개선 단언을 명시적으로 소유한다.
- 상태: 위임 후보

### 비상연락망 — /admin/contacts
- 입력/의존: `/api/contacts`.
- 출력(계약): 이전 연락처 관리 섹션의 비상연락처 CRUD·채널(이메일/문자) 알림 설정 동등 보존.
- 상태: 위임 후보

### 비상 신호 시뮬레이션 — /admin/test-alert
- 입력/의존: `/api/test-alert`.
- 출력(계약): 이전 비상 신호 시뮬레이션 섹션 — 전체 알림 체인을 [테스트] 접두사로 발동하는
  트리거 동작 동등 보존.
- 상태: 위임 후보

### 알림 채널 테스트 발송 — /admin/notify-test
- 입력/의존: `/api/notifications/channels`, `/api/notifications/test`.
- 출력(계약): 이전 채널별 테스트 발송 섹션 — 채널 사용가능 상태 표시(미설정 비활성)·단일 대상
  단건 발송·outcome(sent/failed/not_configured) 표시 동등 보존.
- 상태: 위임 후보

### 시스템 설정 — /admin/system  (현장 정보 + 시스템 설정 합침)
- 입력/의존: `/api/sites`, `/api/settings`, `/api/settings/site_url`.
- 출력(계약): 이전 현장 정보 + 시스템 설정 두 섹션의 동작을 한 페이지에서 동등 보존.
- 상태: 위임 후보

### 사용자 — /admin/users  (계정 관리 + 초대 관리 합침)
- 입력/의존: `/api/invitations` (+ 계정 관리가 쓰던 기존 계약).
- 출력(계약): 이전 계정 관리 + 초대 관리 두 섹션의 동작을 한 페이지에서 동등 보존.
- 상태: 위임 후보

### 저장소 관리 — /admin/storage
- 입력/의존: `/api/storage`, `/api/archives`.
- 출력(계약): 이전 저장소 관리 섹션의 저장소 사용량·아카이브 관리 동작 동등 보존.
- 상태: 위임 후보

### 임시 CCTV 링크 — /admin/cctv-links
- 입력/의존: `/api/links`, `/api/links/temp`.
- 출력(계약): 이전 임시 CCTV 링크 섹션의 링크 생성·목록·만료 동작 동등 보존.
- 상태: 위임 후보

## (횡단) 접합부 계약

단위 사이 공통으로 보장되어야 하는 것 — URL 체계, 딥링크, 뒤로가기=허브 복귀, admin 게이트,
open-redirect 가드 확장 — 은 별도 문서가 소유한다:
**[[admin-routing-navigation-contract]]**. 모든 서브페이지·허브 단위는 이 계약의 인터페이스에만
의존하며, 서로의 내부를 알지 않는다.
