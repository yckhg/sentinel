# 채널별 테스트 발송 스펙

> 상태: 살아있는 계약 (living spec) · 독자: 스펙 작성자 / 오케스트레이터

## 목적 / 의도

- 비상연락망(수신자·채널)을 설정하는 **관리자**가, 저장해 둔 알림 채널이 실제로 동작하는지를 **채널마다 한 번씩 따로** 확인할 수 있게 한다. 이메일은 이메일대로, SMS는 SMS대로 독립적으로 검증한다.
- 관리자 화면에는 이메일 입력 옆의 "테스트 전송" 버튼과, 연락처(전화번호) 옆의 "테스트 전송" 버튼이 있다. 각 버튼은 자기 채널로 **단 1건의 테스트 메시지**만 실제 발송 경로로 보내고 그 시도의 성패를 돌려준다.
- 핵심 의도 세 가지:
  1. **채널 독립** — 이메일 테스트가 SMS를 발송하지 않고, SMS 테스트가 이메일을 발송하지 않는다. 두 트리거는 완전히 분리된다.
  2. **런타임 설정 반영** — 어떤 채널이 사용 가능한지는 **요청을 처리하는 그 시점의 런타임 알림 설정**으로 판정한다(자격증명은 언제든 바뀔 수 있으므로 프로세스 시작 시점 스냅샷에 고정하지 않는다). 미설정 채널의 테스트는 크래시·무한 대기·거짓 성공 주장 없이 명확한 **"미설정(not_configured)"** 결과를 돌려주고, UI는 이를 표시하거나 버튼을 비활성화하며 안내한다.
  3. **정직한 성패 보고** — 설정된 채널의 테스트는 실제 발송 경로로 발송을 시도하고, 그 시도의 성공/실패를 그대로 보고한다(설정되어 있다는 이유만으로 성공을 주장하지 않는다).
- **KakaoTalk 테스트는 이 단위의 범위 밖(연기)** 이다. 이 단위는 이메일·SMS 두 채널만 다룬다.

## 언어 · 런타임

- 관리자 대면 계약 표면은 중앙 REST API(web-backend, Go, JWT admin 인증)와 그 위의 관리자 UI 어포던스(web-frontend, React) 두 층으로 구성된다(식별용 — 구현 지시가 아니다).
- 실제 채널 발송은 시스템의 **운영 발송 경로**(위기 알림이 쓰는 것과 동일한 이메일/SMS 발송 경로)를 통해 이루어진다. 테스트라 해서 별도 목(mock) 경로를 타지 않는다.

## 의존 도구 · 시스템

| 시스템 | 용도 | 계약 소유자 |
|--------|------|-------------|
| 관리자 REST API (web-backend) | 테스트 발송 트리거와 채널 사용가능 상태(status) 읽기의 호스트이자 admin 인증 경계 | `docs/spec/interface-web-api.md` (본 스펙의 "API 계약 델타" 참조) |
| 운영 이메일 발송 경로 | 설정 시 실제 이메일 1건 발송 시도 | 시스템 이메일 채널 (SMTP 등) |
| 운영 SMS 발송 경로 | 설정 시 실제 SMS 1건 발송 시도 | 시스템 SMS 채널 (외부 벤더) |
| 관리자 UI (web-frontend) | 이메일/연락처 옆 "테스트 전송" 버튼 + 미설정 채널 표시·비활성·안내 | `docs/services/web-frontend.md` |
| 런타임 알림 설정 소스 | 채널별 자격증명·활성 여부의 **살아있는** 출처. status 판정과 실제 발송이 **같은 소스**를 참조 | 시스템 알림 설정 |

- 채널 집합은 이 단위에서 정확히 `{email, sms}` 두 종이다. KakaoTalk은 이 집합에 포함되지 않는다.

## 입력

### 1. 채널 사용가능 상태 읽기 (status) — 관리자

- 입력 없음(admin 인증만). in-scope 채널 각각의 사용가능 여부를 조회한다.

### 2. 테스트 이메일 트리거 — 관리자

- 대상 이메일 주소 1개(발송 목적지). 주소가 없거나 형식이 유효하지 않으면 발송 없이 거절(400).

### 3. 테스트 SMS 트리거 — 관리자

- 대상 연락처(또는 전화번호) 1개. 값이 없거나 전화번호 형식(`01[016789]-\d{3,4}-\d{4}`)이 아니면 발송 없이 거절(400).

## 출력 (계약)

1. **채널 상태 계약** — status 읽기는 in-scope 채널(`email`, `sms`) **각각**에 대해 `usable`(boolean)과 미사용 시 사람이 읽을 수 있는 사유(reason)를 제공한다. 응답의 채널 집합에는 **KakaoTalk이 존재하지 않는다**.
2. **테스트 결과 어휘(3종)** — 테스트 이메일·테스트 SMS 트리거의 결과는 정확히 다음 3종 중 하나다:
   - `sent` — 실제 발송 경로로 발송을 시도했고 채널/전송이 수락함(성공).
   - `failed`(+사유) — 실제 발송 경로로 시도했으나 전송/공급자 오류로 실패(설정되어 있어도 성공을 주장하지 않는다).
   - `not_configured` — 해당 채널이 요청 시점 설정상 사용 불가. **발송을 시도하지 않으며 성공을 주장하지 않는다.**
3. **채널 독립** — 테스트 이메일 트리거는 이메일 발송 경로만 시도한다(SMS 발송 0건). 테스트 SMS 트리거는 SMS 발송 경로만 시도한다(이메일 발송 0건). 하나의 트리거가 다른 채널을 부수적으로 발동시키지 않는다.
4. **런타임 설정 판정** — 채널 `usable` 판정과 테스트 시의 미설정/설정 분기는 모두 **요청을 처리하는 시점의 런타임 알림 설정**에서 파생되며, 프로세스 시작 시 캐시한 값이 아니다. 실제 발송 경로가 참조하는 것과 **같은 살아있는 소스**를 status도 참조한다. 따라서 설정을 바꾸면 **재시작 없이** 다음 요청부터 반영된다.
5. **미설정 시 안전 종결(크래시·무한 대기 금지)** — 미설정 채널로 테스트를 요청하면 유한 시간 내에 `not_configured`로 종결한다. 어떤 경우에도 요청이 크래시하거나 무한 대기하거나 조용히 성공을 주장하지 않는다.
6. **유한 지연** — 설정된 채널이라도 대상 공급자/서버가 무응답이면 테스트 요청은 채널 타임아웃 상한 내에 `failed`로 종결되며 무한 대기하지 않는다.
7. **관리자 전용 권한** — status 읽기·테스트 이메일·테스트 SMS 세 표면 모두 admin 권한을 요구한다(연락처 관리 CUD와 **동일 권한**). 비-admin(user) 요청은 `403`, 무인증 요청은 `401`로 거절되며 어떤 발송도 하지 않는다.
8. **KakaoTalk 부재** — 이 단위에는 KakaoTalk을 대상으로 하는 테스트 발송 경로가 **존재하지 않는다**. status 채널 집합에도 없고, KakaoTalk을 지정한 테스트 요청은 지원되지 않는(거절되는) 채널로 취급된다.
9. **UI 어포던스** — 관리자 화면은 이메일 필드 옆과 연락처(전화번호) 옆에 각각 "테스트 전송" 버튼을 제공한다. 채널이 미설정(status `usable=false`)이면 해당 버튼은 비활성화되거나 눌렀을 때 `not_configured`를 사용자에게 안내한다(성공한 것처럼 보이게 하지 않는다).

## 핵심 로직 (동작)

1. status 요청 → 요청 시점 런타임 설정에서 `email`·`sms` 각각의 `usable`/`reason`을 계산해 반환(§출력 1·4). KakaoTalk은 계산·노출 대상이 아님(§출력 8).
2. 테스트 이메일 요청 → 대상 주소 검증(실패 시 400) → 요청 시점 설정으로 이메일 채널 사용가능 판정 → 미설정이면 발송 없이 `not_configured` 반환(§출력 5) → 설정이면 실제 이메일 발송 경로로 1건 시도하고 그 성패를 `sent`/`failed`로 반환(§출력 2). SMS 경로는 건드리지 않음(§출력 3).
3. 테스트 SMS 요청 → 대상 연락처/전화번호 검증(실패 시 400) → 요청 시점 설정으로 SMS 채널 사용가능 판정 → 미설정이면 발송 없이 `not_configured` → 설정이면 실제 SMS 발송 경로로 1건 시도하고 성패 반환. 이메일 경로는 건드리지 않음(§출력 3).
4. 세 표면 모두 진입 시 admin 권한을 강제한다(§출력 7).

## API 계약 델타 (선언 — SSOT 미편집)

> 이 단위는 web-api 접면에 아래 표면을 **추가**한다. 접면 계약의 정본(SSOT)은 `docs/spec/interface-web-api.md`이며, **본 스펙은 그 문서를 편집하지 않고 추가될 계약의 윤곽만 델타로 선언**한다(정본 반영은 오케스트레이터/구현 주체의 몫). 경로명은 대표 형태이며 최종 명명은 SSOT가 소유한다.

- **채널 사용가능 상태 읽기(admin)** — in-scope 채널별 `{channel, usable, reason}` 배열을 반환. `channel ∈ {email, sms}`. KakaoTalk 미포함. (대표: `GET /api/notifications/channels`)
- **채널별 테스트 발송(admin)** — 채널과 대상을 받아 단일 테스트 메시지 1건을 실제 발송 경로로 보내고 `{outcome}` 반환. `outcome ∈ {sent, failed, not_configured}`; `failed`는 사유 동반. `channel ∈ {email, sms}`. (대표: 이메일 `POST /api/notifications/test` `{channel:"email", target:"<email>"}`, SMS `{channel:"sms", target:"<phone|contactId>"}` — 또는 채널별 분리 엔드포인트. 두 트리거는 서로 독립.)
- **권한**: 위 표면 전부 admin 전용(연락처 CUD와 동일 권한). 비-admin `403`, 무인증 `401`.
- KakaoTalk을 지정한 테스트 요청은 지원 채널이 아님(예: `400`/미지원)으로 취급된다.

## 검증 단언 (TDD)

전제: sentinel 스택 기동 중. admin JWT(`ADMIN_TOKEN`)와 user JWT(`USER_TOKEN`)를 확보한다. 테스트는 web-backend(`http://web-backend:8080`, 프록시 경유 시 `/api/...`)로 요청하고, 실제 발송 시도는 운영 이메일/SMS 발송 경로의 로그 또는 목 공급자에서 관측한다. "발송 시도"는 해당 채널 발송 경로에 도달한 1건의 전송 시도를 뜻한다.

- **A (핵심). 설정된 이메일 테스트 → 성공 + 실제 발송 시도**: 이메일 채널이 설정된 상태에서 특정 주소로 테스트 이메일을 요청하면 응답 `outcome`이 `sent`이고, 이메일 발송 경로에 정확히 1건의 발송 시도가 관측된다. 같은 요청으로 SMS 발송 시도는 0건이다. OK: `sent` + 이메일 시도 1건 + SMS 시도 0건. NOK: 그 외.
- **B (핵심). 미설정 이메일 테스트 → not_configured(무발송·무크래시)**: 이메일 채널이 설정되지 않은 상태에서 테스트 이메일을 요청하면 응답 `outcome`이 `not_configured`이고(성공 주장 없음), 이메일 발송 시도 0건이며, 요청은 유한 시간 내에 정상 응답으로 종결된다(크래시·타임아웃 무한대기 없음). OK: `not_configured` + 발송 0건 + 유한 응답. NOK: `sent` 주장·발송 발생·크래시·무응답.
- **C (핵심). SMS 테스트 미설정/설정 분기**: SMS 채널 미설정 상태에서 테스트 SMS 요청 → `not_configured` + SMS 발송 0건 + 크래시 없음. 이어서 SMS 채널을 설정한 뒤 동일 요청 → SMS 발송 경로에 발송 시도 1건이 관측되고 응답은 `sent` 또는 (공급자 오류 시) `failed`다. OK: 미설정=not_configured·무발송, 설정=발송 시도 1건. NOK: 미설정인데 발송 시도 발생·크래시, 또는 설정인데 발송 시도 없음.
- **D (핵심). 런타임 설정 판정(재시작 없이 반영)**: 이메일 채널을 미설정 상태로 두고 status를 읽으면 `email.usable=false`; 설정을 채운 뒤(프로세스 재시작 없이) 곧바로 status를 다시 읽으면 `email.usable=true`로 바뀐다. 반대로 되돌리면 다시 `false`가 된다. 동일하게 테스트 트리거의 `not_configured`↔`sent` 분기도 재시작 없이 뒤바뀐다. OK: 두 연속 요청 사이 설정 변경이 재시작 없이 usability/분기에 반영. NOK: 재시작해야만 반영되거나 시작 시점 값에 고정.
- **E (핵심). 채널 독립(교차 발동 없음)**: 테스트 이메일을 1회 요청하면 이메일 발송 경로에만 시도가 관측되고 SMS 발송 시도는 0건이다. 테스트 SMS를 1회 요청하면 그 반대다. OK: 각 트리거가 자기 채널만 발동. NOK: 한 트리거가 다른 채널 발송을 유발.
- **F (핵심). 관리자 전용 권한**: status·테스트 이메일·테스트 SMS 세 표면 각각에 대해 user JWT로 요청하면 `403`, 무인증(토큰 없음)으로 요청하면 `401`이며, 어느 경우에도 발송 시도가 0건이다. admin JWT로는 정상 처리된다. OK: 비-admin 403 / 무인증 401 / 발송 0건. NOK: 비-admin 또는 무인증이 발송을 유발하거나 200으로 처리됨.
- **G (핵심). KakaoTalk 테스트 경로 부재**: status 응답의 채널 집합에 KakaoTalk이 없다(정확히 `email`·`sms`만). KakaoTalk을 대상으로 지정한 테스트 발송 요청은 지원되지 않는 채널로 거절된다(발송 시도 0건). OK: status에 KakaoTalk 없음 + KakaoTalk 테스트 거절·무발송. NOK: status에 KakaoTalk 노출되거나 KakaoTalk 테스트가 발송을 시도.
- **H (일반). 설정됐어도 실패는 실패로 보고**: 이메일(또는 SMS) 채널이 설정되어 있으나 발송이 공급자/전송 오류로 실패하도록 구성한 뒤 테스트를 요청하면 응답 `outcome`이 `failed`(+사유)이며 `sent`가 아니다. OK: 실패 시 `failed`. NOK: 실제 실패인데 `sent`를 주장.
- **I (일반). 유한 지연(무응답 공급자)**: 설정된 채널의 대상 공급자가 무응답이 되도록 구성한 뒤 테스트를 요청하면, 요청은 채널 타임아웃 상한 내 유한 시간에 `failed`로 종결된다(무한 대기 없음). OK: 상한 내 종결. NOK: 상한을 넘겨 무한 대기.
- **J (일반). 입력 검증**: 대상 주소가 없거나 형식이 유효하지 않은 테스트 이메일 요청은 발송 없이 `400`. 대상 연락처/전화번호가 없거나 전화번호 형식이 아닌 테스트 SMS 요청은 발송 없이 `400`. OK: 잘못된 입력 → 400 + 발송 0건. NOK: 잘못된 입력으로 발송이 발생.
- **K (일반). status 채널 계약**: status 응답은 `email`·`sms` 각각에 대해 `usable`(boolean)과 미사용 시 `reason`을 포함하고, 이 두 채널만 노출한다(G와 정합). OK: 두 채널 각각 usable/reason 제공 + 그 외 채널 없음. NOK: 채널 누락·과잉 노출·usable 필드 부재.

## 검증 스킵 선언

- **SMS 벤더 배관 정합** — 사유: SMS 발송 벤더의 구체 엔드포인트·인증 헤더 규격은 notifier 스펙과 공유되며 벤더 확정 전 잠정적이다. 본 단위는 SMS를 **추상 발송 경로**(설정 시 전화번호 대상 1건 발송 시도 → 성공/실패 판정)로만 검증하고, 특정 벤더 요청 스키마 정합은 지금 검증하지 않는다. · 중요도: 일반 · 해소 조건: SMS 벤더 스펙 확정 시 구체 계약·단언 추가.
