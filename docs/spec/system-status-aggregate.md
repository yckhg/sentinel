# 시스템 상태 집계(현재-상태 요약 창) 스펙

> 상태: 살아있는 계약 (living spec) · 독자: 스펙 작성자 / 오케스트레이터

## 목적 / 의도

Sentinel 운영자가 보는 **시스템 상태 화면**은 시스템의 **지금 이 순간 상태**를 요약해 보여주는 창이다. 전체 장비를 한 줄씩 나열하는 대시보드도, 상태 변화 로그의 덤프도 아니다.

이 창이 보장하는 핵심 의도는 **경계성(boundedness)** 이다. 감시 대상 장비 수가 10대든 1,000만 대든, 운영자에게 돌아오는 응답의 크기와 화면에 그려지는 항목 수는 **같은 자릿수(order of magnitude)** 로 유지된다. 이를 성립시키는 원리는 단순하다 — **정상은 개수로만, 예외만 개별로.**

1. 상태 응답은 **집계(aggregate)** 다: 요약 카운트(정상 N / 이상 M / 오프라인 K) + 감시하는 **서비스 전체**(작은 고정 집합, 각각 alive/down) + **이상·오프라인 장비만** 담은 목록. 정상 장비는 오직 카운트로 대표되며, 한 장비당 한 행으로 나열되지 않는다.
2. 특정 장비의 **현재 상태**(생존 여부 / 마지막 heartbeat 경과)는 나열이 아니라 **검색**으로 조회한다 — 정상 장비라도 검색으로는 찾을 수 있다.
3. 특정 장비의 **상태 전이 이력**은 **드릴다운**이다 — 운영자가 그 장비를 선택했을 때 요청되며, 요약 창의 집계 응답에는 절대 포함되지 않는다.
4. CCTV/카메라의 연결 상태(connected/disconnected)도 이 현재-상태 그림의 일부다.

이 스펙은 그 "현재-상태 요약 창"이 무엇을 보장해야 하는가만 규정한다. 화면 배치·색상·폴링 주기 같은 표현은 구현에 위임한다.

## 언어 · 런타임

- 집계·검색·드릴다운 응답을 생성하는 상태 소유 서비스: Go (표준 `net/http`), 컨테이너 내부 `:8080` 리스닝. 외부 노출은 상위 proxy 책임.
- 요약 창을 렌더링하는 클라이언트: 모바일 우선 React SPA(정적 빌드, nginx 서빙). 백엔드 응답 shape에만 의존하며 백엔드 내부 구현은 알지 못한다.
- (식별용이며 빌드 지시가 아니다.)

## 의존 도구 · 시스템

- **상태 소유 서비스의 영속 저장소** — 장비 등록/`last_seen`/soft-delete 상태, 상태 전이 이력을 소유하는 단일 DB. (스키마 SSOT는 상태 소유 서비스 스펙이 소유.)
- **서비스 생존 감시원** — 감시 대상 동료 서비스의 헬스를 주기 폴링하고 임계 지속 시 상태를 확정하는 감시 루프. 서비스 alive/down 판정의 출처.
- **장비 관측 보고 경로** — H/W↔S/W 접점이 heartbeat/alert를 관측해 장비의 `last_seen`·경보 상태를 갱신하는 경로. 장비 현재-상태의 출처.
- **스트리밍 상태 출처** — 카메라별 연결(connected/disconnected) 상태를 제공하는 경로. 카메라 현재-상태의 출처.
- **런타임 임계값** — 장비를 오프라인으로 판정하는 heartbeat 경과 임계, 서비스를 down으로 확정하는 실패 지속 임계 등은 설정에서 런타임에 재읽기된다(재시작 없이 반영).

동료 출처(스트리밍 상태 등) 중 하나가 죽어도 집계 응답 자체는 성공한다 — 해당 출처가 제공하던 항목은 보수적(연결 끊김/미상) 값으로 표시될 뿐이다.

## 출력 (계약)

### 집계 응답 (요약 창의 단일 조회)

집계 응답은 **크기가 예외 장비 수에만 비례**하는 세 부분으로 구성된다.

1. **요약 카운트(summary)** — 장비 전체를 세 배타 범주로 분류한 개수:
   - `healthy`(정상): 등록·미삭제이며 생존(최근 heartbeat)하고 경보/오류 상태가 아닌 장비 수
   - `abnormal`(이상): 생존하나 경보 발동 중이거나 비정상 동작 상태로 보고된 장비 수
   - `offline`(오프라인): 마지막 heartbeat 경과가 오프라인 임계를 넘은(=미생존) 장비 수
   - 세 카운트의 합은 감시 대상(미삭제) 장비 총수와 일치한다.
2. **서비스 목록(services)** — 감시하는 서비스의 **작은 고정 집합 전체**. 각 항목은 식별자와 `alive|down` 상태를 가진다. 이 목록은 장비 수와 무관하게 항상 완전히 존재한다.
3. **예외 장비 목록(exceptions)** — `abnormal` 또는 `offline`인 장비**만** 개별 항목으로 담는다. 각 항목은 그 장비의 현재-상태 표시에 필요한 필드(식별자/표시명, 범주 또는 상태, 마지막 관측 경과, 이상 사유)를 가진다. **healthy 장비는 이 목록에 절대 나열되지 않는다** — 오직 요약 카운트로만 대표된다.

**경계 불변식(핵심):** 집계 응답의 개별 장비 항목 수는 **정확히 비-정상(abnormal + offline) 장비 수**와 같고, 정상 장비 수가 늘어도 증가하지 않는다. 즉 정상 장비 10대일 때와 10,000,000대일 때(같은 예외 수 전제) 개별 장비 항목 수는 동일하다. 요약 카운트는 O(1) 크기이고, 서비스 목록은 고정 상한 크기다. 따라서 응답 전체 크기는 장비 총수가 아니라 예외 장비 수에 의해서만 결정된다.

**전이-로그 부재 불변식(핵심):** 집계 응답에는 **어떤 장비의 상태 전이 이력(transition-log) 항목도 포함되지 않는다.** 집계는 오직 각 구성요소의 *현재* 스냅샷과 카운트만 담는다.

### 카메라 현재-상태

카메라의 연결 상태(`connected|disconnected`)는 현재-상태 그림의 관측 가능한 일부다. 요약 창은 카메라 연결 상태를 현재-상태 데이터로부터 읽을 수 있다.

### 장비 검색 응답 (현재 상태 조회)

단일 장비 식별자로 조회하면 그 장비의 **현재 상태**(생존 여부 alive/offline, 마지막 heartbeat 경과, 이상 여부)를 반환한다. **대상 장비가 healthy여도 조회된다** — 검색은 나열이 아니므로 정상 장비도 지목해서 찾을 수 있다. 미등록/삭제 장비 조회는 리소스-부재로 응답한다.

### 장비 이력 드릴다운 응답 (요구 시)

단일 장비 식별자로 이력을 조회하면 **그 장비 하나의** 최근 상태 전이 N건(상한 있음)을 반환한다. 다른 장비의 전이는 포함되지 않는다. 이 응답은 집계 응답과 별개의 조회로만 얻어지며, 집계 응답이 이를 대신 담지 않는다.

## 핵심 로직 (동작)

### 장비 범주 판정 (배타·전수)
- 감시 대상은 **미삭제(soft-delete 아님) 장비**다. soft-delete된 장비는 집계·카운트에서 즉시 제외된다.
- 각 미삭제 장비는 `healthy` / `abnormal` / `offline` **정확히 하나**로 분류된다(배타). 세 카운트의 합 = 미삭제 장비 총수(전수).
- `offline`: 마지막 관측(heartbeat) 경과가 오프라인 임계를 초과 → 미생존.
- `abnormal`: 생존(경과가 임계 이내)하나 경보 발동 중이거나 비정상 상태로 보고됨.
- `healthy`: 생존이며 경보/비정상이 아님.
- 판정 임계는 런타임 설정에서 읽으며, 변경은 재시작 없이 이후 집계에 반영된다.

### 정상은 카운트, 예외만 개별
- 집계는 healthy 장비를 개별 항목으로 실체화하지 않는다 — 카운트만 증가시킨다.
- 예외 목록에는 abnormal·offline 장비만 담는다. 이 목록의 길이는 예외 수에 비례하고 정상 수에 불변이다.
- 장비가 아무리 많아도(정상이 대다수인 정상 운영 상태) 응답은 "요약 카운트 + 고정 서비스 목록 + (대개 짧은) 예외 목록"으로 경계 안에 머문다.

### 서비스 목록은 항상 완전
- 감시 서비스 집합은 작고 고정적이다. 집계는 이 집합 전체를 매번 완전히 담으며, 각 서비스의 현재 `alive|down`을 표시한다. 서비스가 하나도 down이 아니어도 목록은 축약되지 않는다(운영자가 "모든 서비스가 살아있음"을 명시적으로 확인 가능).

### 현재 상태 vs 이력의 분리
- 집계·검색은 **현재 스냅샷**만 다룬다.
- **이력(전이 로그)** 은 특정 장비를 지목한 드릴다운 요청으로만 제공된다. 이력은 요청 대상 한 장비로 스코프가 한정되며, 요약 창의 기본 로드에는 어떤 이력도 실려 나가지 않는다.

### 요약 창 렌더링 (클라이언트 계약)
- 요약 창은 (a) 서비스 전체 상태, (b) 장비 요약 카운트, (c) 예외(이상/오프라인) 목록, (d) 특정 장비의 현재 상태를 찾는 **검색**을 제공한다.
- 예외 목록의 한 항목을 선택하면 그 장비의 이력을 드릴다운으로 요청해 표시한다.
- 화면에 그려지는 개별 장비 행 수는 예외 수 + (검색 시) 조회된 1건이며, 정상 장비 수에 비례해 늘어나지 않는다.

## 검증 단언 (TDD)

이하 "집계 조회"는 요약 창이 호출하는 현재-상태 집계 접면, "검색"은 단일 장비 현재-상태 조회 접면, "드릴다운"은 단일 장비 이력 조회 접면을 가리킨다. `D_healthy`/`D_abnormal`/`D_offline`는 각 범주의 장비 수.

- **A. 집계 = 예외만 개별 (핵심).** `D_healthy`대의 정상 장비와 `D_abnormal`대의 이상 + `D_offline`대의 오프라인 장비가 있을 때 집계 조회 → 개별 장비 항목 수가 **정확히 `D_abnormal + D_offline`** (정상 장비 항목 0). 요약 카운트는 `healthy==D_healthy`, `abnormal==D_abnormal`, `offline==D_offline`이며 세 값의 합이 미삭제 장비 총수와 같다.
- **B. 경계 불변식 (핵심).** 예외 수(`D_abnormal + D_offline`)를 고정한 채 정상 장비 수만 바꾼 두 상황 — 예: 정상 10대 vs 정상 10,000대 — 에서 집계 조회의 개별 장비 항목 수가 **동일**하다. 응답의 개별 장비 항목 수는 정상 장비 수에 불변이다.
- **C. 정상 장비는 나열되지 않음 (핵심).** 모든 장비가 healthy(예외 0)인 상태에서 집계 조회 → 개별 장비 항목 수 0, 요약 카운트 `healthy==총수 · abnormal==0 · offline==0`. 어떤 healthy 장비의 식별자도 예외 목록에 등장하지 않는다.
- **D. 검색은 정상 장비도 찾음 (핵심).** healthy로 분류된 특정 장비 식별자로 검색 → 그 장비의 현재 상태(생존 alive, 마지막 heartbeat 경과)를 반환한다(집계 예외 목록엔 없지만 검색으로는 조회됨). 미등록/삭제 장비 식별자로 검색 → 리소스-부재 응답.
- **E. 이력은 드릴다운 전용, 집계엔 전이-로그 없음 (핵심).** 한 장비에 상태 전이가 여러 번 발생한 뒤: (a) 집계 조회 응답에는 어떤 전이-로그 항목도 없다(현재 스냅샷·카운트만). (b) 그 장비 식별자로 드릴다운 조회 → 그 장비의 최근 전이 N건이 반환되고, **다른 장비의 전이는 하나도 포함되지 않는다.**
- **F. 서비스 목록 항상 완전 (핵심).** 집계 조회 응답의 서비스 목록은 감시 대상 고정 집합 **전체**를 담고 각 항목이 `alive|down`을 가진다. 모든 서비스가 alive여도 목록이 축약되지 않는다. 서비스 1개를 내리고 임계 시간 경과 후 재조회 → 그 서비스만 `down`, 나머지는 여전히 목록에 존재하며 `alive`.
- **G. 카메라 연결 상태 관측 가능 (핵심).** 한 카메라가 연결됨(스트리밍 활성)일 때 현재-상태 데이터에서 그 카메라가 `connected`로 관측되고, 스트리밍이 끊긴 카메라는 `disconnected`로 관측된다.
- **H. 오프라인 임계 경계.** 장비의 마지막 heartbeat 경과가 오프라인 임계를 넘으면 다음 집계 조회에서 그 장비가 `offline` 카운트에 포함되고 예외 목록에 등장한다. 임계 이내로 최근 heartbeat가 관측되면 다음 집계에서 `healthy`로 돌아가 예외 목록에서 사라진다(카운트로만 대표).
- **I. soft-delete 제외.** soft-delete된 장비는 집계의 어떤 카운트에도 포함되지 않고 예외 목록에도 나타나지 않는다(세 카운트의 합이 미삭제 장비 총수와 계속 일치). 이후 관측 보고로 복원되면 다시 카운트 대상이 된다.
- **J. 요약 창 화면 경계.** 요약 창을 정상 장비가 대다수인 상태로 렌더링하면 화면의 개별 장비 행 수가 예외 수 + (검색 시) 1이며, 정상 장비 수에 비례해 증가하지 않는다. 서비스 상태 전체·장비 요약 카운트·예외 목록·장비 검색 입력이 화면에 모두 존재한다.

## API 접면 변경 노트 (선언적 — SSOT 미편집)

> 이 절은 접면 SSOT(`docs/spec/interface-web-api.md`)를 **본 스펙에서 편집하지 않는다.** 아래는 그 SSOT가 반영해야 할 계약 델타의 선언적 요약이며, 실제 SSOT 정합은 오케스트레이터/설계 세션의 몫이다.

- **집계 헬스 접면**은 "모든 엔티티를 한 항목씩 나열한 배열"이 아니라 **요약 카운트 + 고정 서비스 목록 + 예외(이상/오프라인) 장비 목록**으로 이루어진 **집계**를 계약한다. 개별 장비 항목 수는 예외 수에 비례하며 정상 장비 수에 불변이다(경계 불변식). 정상 장비는 카운트로만 대표되고 개별 항목으로 나열되지 않는다.
- **장비 현재-상태 검색**은 집계와 별개의 접면으로, 단일 장비 식별자를 받아 그 장비의 현재 상태(생존/마지막 관측 경과/이상 여부)를 반환한다 — 정상 장비도 조회 대상.
- **장비 이력 드릴다운**은 또 다른 별개 접면으로, 단일 장비 식별자를 받아 그 장비 하나의 최근 상태 전이 N건만 반환한다. 이력은 집계 응답에 실리지 않는다.
- **카메라 연결 상태**(`connected|disconnected`)는 현재-상태 데이터의 관측 가능한 일부로 유지된다.
