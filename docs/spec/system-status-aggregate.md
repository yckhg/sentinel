# 시스템 상태 집계(현재-상태 요약 창) 스펙

> 상태: 살아있는 계약 (living spec) · 독자: 스펙 작성자 / 오케스트레이터

## 목적 / 의도

Sentinel 운영자가 보는 **시스템 상태 화면**은 **현재-상태 요약 창**이다 — 화면에 담기는 것은 **요약 카운트 + 고정 서비스 상태 + 예외(이상/오프라인) 장비만**이다. 보강하자면, 이 창은 전체 장비를 한 줄씩 나열하는 대시보드도, 상태 변화 로그의 덤프도 아니다. 이 부정(전수 나열 아님 · 로그 덤프 아님)은 아래 **경계 불변식**과 **healthy 장비 미나열** 규정을 강제하는 load-bearing 정의이므로 그대로 유지한다.

이 창이 보장하는 핵심 의도는 **경계성(boundedness)** 이다. 감시 대상 장비 수가 10대든 1,000만 대든, 운영자에게 돌아오는 응답의 크기와 화면에 그려지는 항목 수는 **같은 자릿수(order of magnitude)** 로 유지된다. 이를 성립시키는 원리는 단순하다 — **정상은 개수로만, 예외만 개별로.**

1. 상태 응답은 **집계(aggregate)** 다: 요약 카운트(정상 N / 이상 M / 오프라인 K) + 감시하는 **서비스 전체**(작은 고정 집합, 각각 `healthy|unhealthy` — 계약 12 어휘) + **이상·오프라인 장비만** 담은 목록. 정상 장비는 오직 카운트로 대표되며, 한 장비당 한 행으로 나열되지 않는다.
2. 특정 장비의 **현재 상태**(생존 여부 / 마지막 heartbeat 경과)는 나열이 아니라 **검색**으로 조회한다 — 정상 장비라도 검색으로는 찾을 수 있다.
3. 특정 장비의 **상태 전이 이력**은 **드릴다운**이다 — 운영자가 그 장비를 선택했을 때 요청되며, 요약 창의 집계 응답에는 절대 포함되지 않는다.
4. CCTV/카메라의 연결 상태(connected/disconnected)도 이 현재-상태 그림의 일부다.

이 스펙은 그 "현재-상태 요약 창"이 무엇을 보장해야 하는가만 규정한다. 화면 배치·색상·폴링 주기 같은 표현은 구현에 위임한다.

## 언어 · 런타임

- 집계·검색·드릴다운 응답을 생성하는 상태 소유 서비스: Go (표준 `net/http`), 컨테이너 내부 `:8080` 리스닝. 외부 노출은 상위 proxy 책임.
- 요약 창을 렌더링하는 클라이언트: 모바일 우선 React SPA(정적 빌드, nginx 서빙). 백엔드 응답 shape에만 의존하며 백엔드 내부 구현은 알지 못한다. 시스템상태 패널은 집계를 **신규 엔드포인트 `GET /api/health/summary`**에서 소비하고, 카메라 연결 상태는 **별도로 `GET /api/cameras`(계약 3)**를 조회한다(집계 응답 shape 밖).
- (식별용이며 빌드 지시가 아니다.)

## 의존 도구 · 시스템

- **상태 소유 서비스의 영속 저장소** — 장비 등록/`last_seen`/soft-delete 상태, 상태 전이 이력을 소유하는 단일 DB. (스키마 SSOT는 상태 소유 서비스 스펙이 소유.)
- **서비스 생존 감시원** — 감시 대상 동료 서비스의 헬스를 주기 폴링하고 임계 지속 시 상태를 확정하는 감시 루프. 서비스 `healthy|unhealthy` 판정의 출처(계약 12).
- **장비 관측 보고 경로** — H/W↔S/W 접점이 heartbeat/alert를 관측해 장비의 `last_seen`·경보 상태를 갱신하는 경로. 장비 현재-상태의 출처.
- **스트리밍 상태 출처** — 카메라별 연결(connected/disconnected) 상태를 제공하는 경로. 카메라 현재-상태의 출처.
- **런타임 임계값** — 장비를 오프라인으로 판정하는 heartbeat 경과 임계, 서비스를 down으로 확정하는 실패 지속 임계 등은 설정에서 런타임에 재읽기된다(재시작 없이 반영).

동료 출처(스트리밍 상태 등) 중 하나가 죽어도 집계 응답 자체는 성공한다 — 해당 출처가 제공하던 항목은 보수적(연결 끊김/미상) 값으로 표시될 뿐이다.

## 출력 (계약)

### 집계 응답 (요약 창의 단일 조회 — `GET /api/health/summary` 신규)

집계는 **신규 엔드포인트 `GET /api/health/summary`**가 계약한다. 기존 계약 12의 평면 배열 `GET /api/health`·`GET /api/health/events`는 변경 없이 존속하며(공존), 본 집계 응답은 그 위에 요약 shape를 신설한다. 집계 응답은 **크기가 예외 장비 수에만 비례(상한 내)**하는 세 부분으로 구성된다.

1. **요약 카운트(summary)** — 장비 전체를 세 배타 범주로 분류한 개수:
   - `healthy`(정상): 등록·미삭제이며 생존(최근 heartbeat)하고 경보 상태가 아닌 장비 수
   - `abnormal`(이상): **생존(alive)이며 `alertState == "active"`**인 장비 수 (계약 6의 `alertState` 어휘 그대로 사용 — 스키마에 없는 "비정상 동작 상태"는 정의에 넣지 않는다)
   - `offline`(오프라인): 마지막 heartbeat 경과가 오프라인 임계를 넘은(=미생존) 장비 수
   - 세 카운트의 합은 감시 대상(미삭제) 장비 총수와 일치한다.
2. **서비스 목록(services)** — 감시하는 서비스의 **작은 고정 집합 전체**. 각 항목은 식별자와 `healthy|unhealthy` 상태를 가진다(계약 12 어휘와 통일 — 패널이 alive/down으로 표기한다면 `healthy→alive`·`unhealthy→down` 1:1 매핑임을 명시). 감시 대상 고정 집합의 SSOT는 **계약 12의 서비스 산출**(docker healthcheck 대상 − {web-backend, mosquitto})이다. 이 목록은 장비 수와 무관하게 항상 완전히 존재한다.
3. **예외 장비 목록(exceptions)** — `abnormal` 또는 `offline`인 장비**만** 개별 항목으로 담되, **명시적 상한(cap, 기본 50건)**을 둔다. 각 항목은 그 장비의 현재-상태 표시에 필요한 필드(식별자/표시명, 범주 또는 상태, 마지막 관측 경과, 이상 사유)를 가진다. 예외 수가 상한을 초과하면 목록은 상한 건까지만 담고 응답에 **오버플로 표식 `exceptionsOverflow: <목록에 담기지 못한 남은 예외 수>`**(예외가 상한 이하면 `0`)를 포함한다. **healthy 장비는 이 목록에 절대 나열되지 않는다** — 오직 요약 카운트로만 대표된다.

**경계 불변식(핵심):** 집계 응답의 개별 장비 항목 수는 **`min(비-정상(abnormal + offline) 장비 수, 상한)`**과 같고, 정상 장비 수가 늘어도 증가하지 않는다. 즉 정상 장비 수를 바꿔도(같은 예외 수 전제) 개별 장비 항목 수는 동일하다. hw-gateway 다운으로 전 장비가 offline이 되는 전면 장애에서도 개별 항목 수는 상한을 넘지 못하고 `exceptionsOverflow`가 나머지를 대표한다. 따라서 **응답 크기 = min(예외 수, 상한) + 요약(요약 카운트 O(1) + 고정 상한 서비스 목록)**으로 경계된다 — 응답 크기는 장비 총수가 아니라 (상한으로 캡된) 예외 장비 수에 의해서만 결정된다. 집계 자체는 SQL COUNT/GROUP BY로 산출되므로 정상 장비를 개별 실체화하지 않는다.

**전이-로그 부재 불변식(핵심):** 집계 응답에는 **어떤 장비의 상태 전이 이력(transition-log) 항목도 포함되지 않는다.** 집계는 오직 각 구성요소의 *현재* 스냅샷과 카운트만 담는다.

### 카메라 현재-상태 (계약 3에서 관측 — 본 스펙 신규 계약 아님)

카메라의 연결 상태(`connected|disconnected`)는 현재-상태 그림의 관측 가능한 일부이나 **본 집계 응답 shape 밖**이다. 요약 창은 카메라 연결 상태를 **기존 계약 3 `GET /api/cameras`의 `status`(`connected|disconnected`)**에서 별도 조회로 읽는다. 본 스펙은 카메라용 신규 접면을 만들지 않으며, 이 데이터의 계약 SSOT는 계약 3이다.

### 장비 검색 응답 (현재 상태 조회 — 계약 6 단건 조회 재사용)

장비 현재-상태 검색은 별도 신접면을 만들지 않고 **계약 6 기반 `GET /api/devices/{id}` 단건 조회**(델타 노트로 계약 6에 단건 GET 추가)로 수행한다. 반환된 device 오브젝트(`lastSeen`·`alertState` 등)에 **집계와 공용인 범주 파생 함수**(같은 임계·같은 `alertState=="active"` 판정)를 적용해 그 장비의 **현재 상태**(생존 여부 alive/offline, 마지막 heartbeat 경과, 이상 여부)를 도출한다. **대상 장비가 healthy여도 조회된다** — 검색은 나열이 아니므로 정상 장비도 지목해서 찾을 수 있다. 미등록/삭제(`deletedAt`) 장비 조회는 계약 6 규약대로 `404`(리소스-부재)로 응답한다.

### 장비 이력 드릴다운 응답 (요구 시 — 계약 12 events + `entity_id` 필터)

장비별 상태 전이 이력은 **기존 `GET /api/health/events`에 `entity_id` 필터를 추가**(델타 노트)해 조회한다. `health_events`는 감시 대상의 **healthy↔offline(=unhealthy) 전이만** 기록하므로, 이력 드릴다운은 **그 장비의 online/offline 전이 이력** 최근 N건(상한 있음)을 반환하며 **다른 장비의 전이는 하나도 포함되지 않는다**. 경보(alert)의 발동/해제 전이 이력은 `health_events` 소관이 아니라 **incident 소관**이므로 본 드릴다운(E) 범위 밖이다. 이 응답은 집계 응답과 별개의 조회로만 얻어지며, 집계 응답이 이를 대신 담지 않는다.

## 핵심 로직 (동작)

### 장비 범주 판정 (배타·전수)
- 감시 대상은 **미삭제(soft-delete 아님) 장비**다. soft-delete된 장비는 집계·카운트에서 즉시 제외된다.
- 각 미삭제 장비는 `healthy` / `abnormal` / `offline` **정확히 하나**로 분류된다(배타). 세 카운트의 합 = 미삭제 장비 총수(전수).
- `offline`: 마지막 관측(heartbeat) 경과가 오프라인 임계를 초과 → 미생존. 오프라인 임계는 **`health.sensor_alive_threshold_sec`(계약 11)**를 재사용한다(센서 생존 판정과 동일 임계).
- `abnormal`: **생존(경과가 임계 이내)이며 `alertState == "active"`**(계약 6). 그 외 "비정상 상태 보고" 같은 스키마 밖 신호는 판정에 쓰지 않는다.
- `healthy`: 생존이며 `alertState != "active"`.
- 판정 임계는 런타임 설정에서 읽으며, 변경은 재시작 없이 이후 집계에 반영된다.

### 단일 일관 스냅샷
- 요약 카운트와 exceptions 목록은 **단일 일관 스냅샷(단일 트랜잭션/쿼리)**에서 산출된다. 카운트와 예외 목록이 서로 다른 시점을 보지 않으므로, 관측 보고(`seen`)/soft-delete가 동시에 갱신되는 중에도 **세 카운트의 합 = 미삭제 장비 총수** 불변식(단언 A/I)과 "예외 목록 = 비-정상 장비" 대응이 각 응답 안에서 정합하게 성립한다.

### 정상은 카운트, 예외만 개별
- 집계는 healthy 장비를 개별 항목으로 실체화하지 않는다 — 카운트는 **집합 연산(SQL COUNT/GROUP BY)**으로 산출한다. 카운트 산출 비용은 정상 장비 수에 비례할 수 있으나 응답·렌더 크기는 그렇지 않다(경계 불변식은 **응답 크기·렌더 항목 수**에 한정된다).
- 예외 목록에는 abnormal·offline 장비만 담되 **상한(cap)**으로 캡한다. 목록 길이는 `min(예외 수, 상한)`이고 정상 수에 불변이며, 초과분은 `exceptionsOverflow`가 대표한다.
- 장비가 아무리 많아도(정상이 대다수인 정상 운영 상태) 응답은 "요약 카운트 + 고정 서비스 목록 + (상한 내) 예외 목록"으로 경계 안에 머문다.

### 서비스 목록은 항상 완전
- 감시 서비스 집합은 작고 고정적이다. 이 **기대 서비스 집합의 SSOT는 계약 12의 서비스 산출**(docker healthcheck 대상 − {web-backend, mosquitto})이다. 집계는 이 집합 전체를 매번 완전히 담으며(즉 **기대 서비스 집합 ⊆ 응답 서비스 집합**), 각 서비스의 현재 `healthy|unhealthy`(계약 12 어휘)를 표시한다. 서비스가 하나도 unhealthy가 아니어도 목록은 축약되지 않는다(운영자가 "모든 서비스가 살아있음"을 명시적으로 확인 가능).

### 현재 상태 vs 이력의 분리
- 집계·검색은 **현재 스냅샷**만 다룬다.
- **이력(전이 로그)** 은 특정 장비를 지목한 드릴다운 요청으로만 제공된다. 이력은 요청 대상 한 장비로 스코프가 한정되며, 요약 창의 기본 로드에는 어떤 이력도 실려 나가지 않는다.

### 요약 창 렌더링 (클라이언트 계약)
- 요약 창은 (a) 서비스 전체 상태, (b) 장비 요약 카운트, (c) 예외(이상/오프라인) 목록, (d) 특정 장비의 현재 상태를 찾는 **검색**을 제공한다.
- 예외 목록의 한 항목을 선택하면 그 장비의 이력을 드릴다운으로 요청해 표시한다.
- 화면에 그려지는 개별 장비 행 수는 예외 수 + (검색 시) 조회된 1건이며, 정상 장비 수에 비례해 늘어나지 않는다.

## 검증 단언 (TDD)

이하 "집계 조회"는 요약 창이 호출하는 현재-상태 집계 접면, "검색"은 단일 장비 현재-상태 조회 접면, "드릴다운"은 단일 장비 이력 조회 접면을 가리킨다. `D_healthy`/`D_abnormal`/`D_offline`는 각 범주의 장비 수.

- **A. 집계 = 예외만 개별, 합 불변식 (핵심).** `D_healthy`대의 정상 장비와 `D_abnormal`대의 이상 + `D_offline`대의 오프라인 장비가 있을 때(단 `D_abnormal + D_offline ≤ 상한`) 집계 조회 → 개별 장비 항목 수가 **정확히 `D_abnormal + D_offline`** (정상 장비 항목 0), `exceptionsOverflow == 0`. 요약 카운트는 `healthy==D_healthy`, `abnormal==D_abnormal`, `offline==D_offline`이며 **세 값의 합이 미삭제 장비 총수와 같다**(단일 스냅샷 산출이므로 관측 보고/soft-delete 동시 갱신 중에도 각 응답 안에서 합이 정합).
- **B. 스케일 불변 스모크 (비-핵심 — A에 흡수됨).** 예외 수(`D_abnormal + D_offline`, 상한 이하)를 고정한 채 정상 장비 수만 바꾼 두 상황 — 예: 정상 **2대 vs 정상 200대**(대량 주입 없는 작은-N 스모크) — 에서 집계 조회의 개별 장비 항목 수가 **동일**하다. A+C가 이 불변을 이미 함의하므로 스케일 스모크로만 유지(핵심 아님).
- **C. 정상 장비는 나열되지 않음 (핵심).** 모든 장비가 healthy(예외 0)인 상태에서 집계 조회 → 개별 장비 항목 수 0, 요약 카운트 `healthy==총수 · abnormal==0 · offline==0`. 어떤 healthy 장비의 식별자도 예외 목록에 등장하지 않는다.
- **D. 검색은 정상 장비도 찾음 (핵심).** healthy로 분류된 특정 장비를 **계약 6 `GET /api/devices/{id}`** 단건 조회 → 그 device 오브젝트에 공용 범주 파생 함수를 적용해 현재 상태(생존 alive, 마지막 heartbeat 경과, 이상 여부)를 얻는다(집계 예외 목록엔 없지만 검색으로는 조회됨). 미등록/삭제(`deletedAt`) 장비 식별자로 조회 → `404`(계약 6 규약).
- **E. 이력은 드릴다운 전용, 집계엔 전이-로그 없음 (핵심).** 한 장비에 online/offline 전이가 여러 번 발생한 뒤: (a) 집계 조회 응답에는 어떤 전이-로그 항목도 없다(현재 스냅샷·카운트만). (b) 그 장비 식별자로 **`GET /api/health/events?entity_id=<siteId:deviceId>`** 조회 → 그 장비의 **online/offline 전이** 최근 N건이 반환되고, **다른 장비의 전이는 하나도 포함되지 않는다**(타 장비 전이 0건). 경보(alert) 전이 이력은 `health_events` 소관이 아니므로(incident 소관) 이 응답 범위 밖이다.
- **F. 서비스 목록 항상 완전 (핵심).** 집계 조회 응답의 서비스 목록은 **기대 서비스 집합(계약 12: docker healthcheck − {web-backend, mosquitto}) ⊆ 응답 서비스 집합**이 성립하도록 그 집합 전체를 담고, 각 항목이 `healthy|unhealthy`를 가진다. 모든 서비스가 healthy여도 목록이 축약되지 않는다. 서비스 1개를 내리고 임계(`health.service_down_threshold_sec`) 경과 후 재조회 → 그 서비스만 `unhealthy`, 나머지는 여전히 목록에 존재하며 `healthy`.
- **G. 카메라 연결 상태 관측 가능 (핵심 — 계약 3에서 관측).** 한 카메라가 연결됨(스트리밍 활성)일 때 **계약 3 `GET /api/cameras`의 `status`**에서 그 카메라가 `connected`로 관측되고, 스트리밍이 끊긴 카메라는 `disconnected`로 관측된다. (본 스펙 신규 계약이 아니라 계약 3에서 읽는다 — 집계 응답 shape 밖.)
- **H. 오프라인 임계 경계 + 런타임 반영.** 장비의 마지막 heartbeat 경과가 오프라인 임계(`health.sensor_alive_threshold_sec`)를 넘으면 다음 집계 조회에서 그 장비가 `offline` 카운트에 포함되고 예외 목록에 등장한다. 임계 이내로 최근 heartbeat가 관측되면 다음 집계에서 `healthy`로 돌아가 예외 목록에서 사라진다(카운트로만 대표). **런타임 반영**: 계약 11 `PUT /api/settings`로 임계를 낮추면(재시작 없이) 이전에 healthy였던 장비가 경과가 새 임계를 초과하게 되어 **다음 집계에서 곧바로 offline으로 전이**한다.
- **I. soft-delete 제외 + 단일 스냅샷 정합.** soft-delete된 장비는 집계의 어떤 카운트에도 포함되지 않고 예외 목록에도 나타나지 않는다(세 카운트의 합이 미삭제 장비 총수와 계속 일치). 이후 관측 보고로 복원되면 다시 카운트 대상이 된다. 카운트와 예외 목록이 **단일 일관 스냅샷**에서 산출되므로, soft-delete/복원이 동시에 진행되는 중에 얻은 각 응답 안에서도 합 불변식이 깨지지 않는다.
- **J. 요약 창 화면 경계.** 요약 창을 정상 장비가 대다수인 상태로 렌더링하면 화면의 개별 장비 행 수가 `min(예외 수, 상한)` + (검색 시) 1이며, 정상 장비 수에 비례해 증가하지 않는다(예외가 상한 초과면 오버플로 표식으로 대표). 서비스 상태 전체·장비 요약 카운트·예외 목록·장비 검색 입력이 화면에 모두 존재한다.
- **K. Graceful degradation (동료 출처 정지).** 카메라 연결 상태를 제공하는 동료 출처(스트리밍 등)를 정지시킨 격리 fixture에서: 집계 조회는 여전히 `200`으로 성공하고, 해당 출처가 제공하던 카메라의 연결 상태는 보수적으로 `disconnected`로 관측된다(집계 응답이 그 출처 실패로 전체 실패하지 않는다).

## API 접면 변경 노트 (선언적 — SSOT 미편집)

> 이 절은 접면 SSOT(`docs/spec/interface-web-api.md`)를 **본 스펙에서 편집하지 않는다.** 아래는 그 SSOT가 반영해야 할 계약 델타의 선언적 요약이며, 실제 SSOT 정합은 오케스트레이터/설계 세션의 몫이다.

- **[신설·공존] 집계 헬스 접면 `GET /api/health/summary` 신규.** 요약 카운트 + 고정 서비스 목록 + 예외(이상/오프라인) 장비 목록으로 이루어진 **집계**를 계약한다. 기존 계약 12의 **`GET /api/health`(평면 배열)·healthy↔unhealthy WS push(`system_alarm`)·`GET /api/health/events`·재접속 admin unhealthy 스냅샷 재전달은 변경 없이 존속**한다(회귀 범위 밖 — 계약 12의 기존 불변식은 그대로 보존). 프론트 시스템상태 패널은 `/api/health/summary`를 소비한다.
- **[신설 응답 필드] exceptions 상한 + 오버플로 표식.** `/api/health/summary`의 exceptions 배열은 명시적 상한(기본 50건)을 가지며, 초과 시 응답에 `exceptionsOverflow: <남은 예외 수>`(상한 이하면 `0`)를 포함한다. 개별 장비 항목 수는 `min(예외 수, 상한)`으로 캡되고 정상 장비 수에 불변이다(경계 불변식). 서비스 상태 어휘는 계약 12와 통일해 `healthy|unhealthy`, 감시 서비스 집합의 SSOT는 계약 12의 산출(docker healthcheck − {web-backend, mosquitto}). 장비 offline 판정 임계는 계약 11 `health.sensor_alive_threshold_sec` 재사용.
- **[델타 → 계약 6] 장비 현재-상태 단건 조회 `GET /api/devices/{id}` 추가.** 단일 장비 식별자를 받아 그 device 오브젝트를 반환하고(패널은 공용 범주 파생 함수로 현재 상태를 도출), 미등록/삭제(`deletedAt`) → `404`(계약 6 규약). 별도 신접면을 만들지 않고 계약 6에 단건 GET을 더한다.
- **[델타 → 계약 12] `GET /api/health/events`에 `entity_id` 필터 추가.** 특정 장비 식별자(`siteId:deviceId`)로 그 장비의 online/offline(healthy↔unhealthy) 전이 이력만 조회한다. `health_events`는 online/offline 전이만 기록하므로 경보(alert) 전이 이력은 이 접면 범위 밖(incident 소관). 이력은 집계 응답에 실리지 않는다.
- **[불변 — 계약 3] 카메라 연결 상태**(`connected|disconnected`)는 기존 계약 3 `GET /api/cameras`의 `status`에서 읽으며, **본 집계 응답 shape 밖**이다. 본 스펙은 카메라용 신규 계약을 만들지 않는다(패널이 별도로 `/api/cameras` 조회).
