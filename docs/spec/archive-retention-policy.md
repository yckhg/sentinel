# 아카이브 보존 정책 스펙

> 상태: 살아있는 계약 (living spec) · 독자: 스펙 작성자 / 오케스트레이터
> 단위: recording 서비스 — **영구 아카이브(`archives-data`) 수명주기 계약**
> 관련: `docs/services/recording.md`(서비스 구현 가이드), `docs/spec/archive-download-ux.md`(아카이브 소비 UX), GH #114.
> 접합부: 아카이브 삭제 부작용(보호 세그먼트 해제·메타 정합)은 기존 아카이브 삭제 계약(`DELETE /api/archives/*`)이 소유하는 부작용과 동일 계약을 재사용한다.

## 목적 / 의도

영구 아카이브 볼륨(`archives-data`)이 **무한 증가하지 않도록** 보존 상한을 계약으로 고정한다. recording 서비스는 사고(incident)마다 무재인코딩(`-c copy`) MP4를 영구 적재하며 개당 1~3GB에 이르므로, 자동 축출(eviction) 없이는 단일 온프렘 미니PC의 디스크가 고갈된다. 이 단위는 세 가지를 보장한다.

1. **용량 상한(주 정책)**: 축출 대상 아카이브의 총 용량이 `ARCHIVE_MAX_BYTES`를 넘지 않도록, 초과 시 **오래된 것부터(oldest-first)** 삭제한다.
2. **나이 상한(보조 정책)**: `ARCHIVE_RETENTION_DAYS`를 초과한 아카이브는 용량이 상한 이내여도 축출한다.
3. **수동 아카이브 보호**: 사용자가 명시적으로 생성한 아카이브(`manual_*`)는 어떤 자동 축출로도 삭제되지 않는다.

이 계약은 기존 `recordings-data`의 롤링 청소(`ROLLING_WINDOW_MINUTES` 초과 미보호 세그먼트 자동 삭제)와 **대칭**이다 — 롤링 청소가 임시 `.ts`를 시간창으로 관리하듯, 본 계약은 영구 MP4를 용량·나이로 관리한다. 삭제는 여전히 안전 해체(보호 세그먼트 해제 + 메타 정합)를 수반한다.

## 언어 · 런타임

- **서버**: Go (표준 `net/http`), recording 서비스 프로세스.
- **직렬화**: 파일시스템(MP4 디렉터리 + 메타), REST 관측면은 JSON.

## 의존 도구 · 시스템

- **파일시스템 볼륨 `archives-data`** — 아카이브 디렉터리 집합. 각 아카이브는 `{type}_{...}` 이름을 가지며 `type ∈ {incident, manual}`.
- **주기 스케줄러** — 요청 없이도 축출을 반복 실행하는 서비스 내부 주기 실행 경계(롤링 청소와 동일 성격).
- **아카이브 삭제 부작용 계약** — 아카이브 1건 삭제 시 (a) 그 아카이브가 보호하던 원본 세그먼트의 보호 해제, (b) 아카이브 메타(`metadata.json` 등 목록 SSOT)에서 참조 제거. 본 계약은 이 부작용을 **그대로 재사용**한다(새 삭제 의미를 만들지 않는다).
- **관측면 REST** — `GET /api/archives`(목록), `GET /api/storage`(사용량). 축출 결과가 이 응답에 반영된다.

## 입력

| 설정 | 의미 | 기본값 | 비활성 조건 |
|------|------|--------|-------------|
| `ARCHIVE_MAX_BYTES` | 축출 대상 아카이브 총 용량 상한(바이트). 초과분은 oldest-first 삭제. | 운영자 튜닝값(디스크 여유 대비 산정, 예: 100GiB) | `0` 또는 미설정 → 용량 축출 비활성 |
| `ARCHIVE_RETENTION_DAYS` | 아카이브 최대 보존 일수. 초과 아카이브는 용량 무관 축출. | 미설정(보조 정책, 선택) | `0` 또는 미설정 → 나이 축출 비활성 |

- 대상 집합: `archives-data`의 아카이브 디렉터리들. **용량·나이 산정 및 삭제 대상은 `incident_*`만** 포함하며 `manual_*`은 제외한다.
- 아카이브의 "나이"는 그 아카이브의 생성/기록 시각을 기준으로 한다(관측 가능한 파일 시각).

## 출력 (계약)

한 번의 축출 주기(cycle)가 끝난 뒤 다음 상태가 보장된다:

- **용량 불변식**: `ARCHIVE_MAX_BYTES > 0`이면, `incident_*` 아카이브의 총 용량 ≤ `ARCHIVE_MAX_BYTES`.
- **나이 불변식**: `ARCHIVE_RETENTION_DAYS > 0`이면, 나이가 `ARCHIVE_RETENTION_DAYS`를 초과한 `incident_*` 아카이브가 0건.
- **수동 보존 불변식**: 축출 전 존재하던 모든 `manual_*` 아카이브가 축출 후에도 그대로 존재한다.
- **순서 불변식**: 용량 초과로 삭제가 일어났다면, 살아남은 `incident_*` 집합은 삭제된 것들보다 **더 최근**이다(가장 오래된 것부터 삭제).
- **정합 불변식**: 삭제된 아카이브는 목록 SSOT(`GET /api/archives`)에 더 이상 나타나지 않으며, 메타에 그 아카이브를 가리키는 **댕글링 참조가 없다**. 삭제된 아카이브가 보호하던 세그먼트는 보호 상태로 영구 잔존하지 않는다.
- **무해 불변식**: 상한 이내이고 나이 초과 아카이브가 없으면, 축출 주기는 **아무것도 삭제하지 않는다**(no-op).

## 핵심 로직 (동작)

- **주기 실행**: 축출은 외부 요청 없이 서비스 내부에서 주기적으로 반복된다(롤링 청소와 동일 성격). 초과 상태는 다음 주기 안에 자기수복(self-heal)된다.
- **용량 우선(주 정책)**: `incident_*` 총 용량이 `ARCHIVE_MAX_BYTES`를 초과하면, 가장 오래된 `incident_*`부터 삭제해 총 용량을 상한 이하로 되돌린다.
- **나이(보조 정책)**: `ARCHIVE_RETENTION_DAYS`를 초과한 `incident_*`는 용량이 상한 이내여도 삭제한다. 두 정책이 함께 켜져 있으면 합집합으로 축출한다(나이 초과분 + 용량 초과분).
- **수동 제외**: `manual_*`은 용량 산정·나이 산정·삭제 대상 어디에도 포함되지 않는다. 수동 아카이브만으로 볼륨이 상한을 넘어도 자동 삭제하지 않는다(사용자 명시 산출물 보호).
- **안전 해체**: 삭제는 개별 아카이브 삭제 계약(보호 세그먼트 해제 + 메타 참조 제거)을 재사용한다. 부분 실패 시에도 메타와 실제 디렉터리가 어긋난 댕글링 상태를 남기지 않는다.
- **비활성 반영**: 두 임계값은 각각 독립적으로 끌 수 있다(`0`/미설정). 둘 다 비활성이면 축출은 일어나지 않는다(기존 무한 적재 동작과 동일 — 단, 이는 명시적 비활성 선택일 때만).

## 검증 단언 (TDD)

> 모두 mutating(아카이브 시딩·삭제) — 격리 스택에서 판정(아래 스킵 선언 참조). 시딩은 크기·시각·`type` 접두어를 제어한 더미 아카이브로 구성한다.

- **A (핵심)** — 용량 상한: `ARCHIVE_MAX_BYTES = X`인 스택에서 `incident_*` 총 용량이 `X`를 초과하도록 시딩 → 축출 1주기 → 이후 `incident_*` 총 용량 ≤ `X`. (`GET /api/storage` 또는 볼륨 실측으로 관측)
- **B (핵심)** — 수동 보존: `manual_*` 아카이브 1건 + 상한 초과 `incident_*` 다수를 시딩 → 축출 1주기 → 그 `manual_*`이 여전히 존재하고 `GET /api/archives`에 남아 있다.
- **C (핵심)** — oldest-first 순서: 서로 다른 시각의 `incident_*` N건을 상한 초과로 시딩 → 축출 1주기 → 살아남은 집합은 **가장 최근** 아카이브들이고, 가장 오래된 것들이 삭제되었다(생성시각 오름차순으로 삭제 경계가 형성된다).
- **D** — 나이 상한: `ARCHIVE_RETENTION_DAYS = R` 설정, 용량은 상한 이내로 두고 나이가 `R`을 초과한 `incident_*` 1건을 시딩 → 축출 1주기 → 그 아카이브가 삭제되고, `R` 이내 아카이브는 보존된다.
- **E (핵심)** — 무해(no-op): 용량이 상한 이내이고 나이 초과 아카이브가 없는 상태 → 축출 1주기 → `GET /api/archives` 목록과 총 용량이 **주기 전후 동일**(아무것도 삭제되지 않음).
- **F (핵심)** — 정합: A/C에서 삭제가 발생한 뒤, 삭제된 아카이브 id가 `GET /api/archives` 응답에 없고(댕글링 참조 없음), 목록 SSOT와 실제 디렉터리 집합이 일치한다.
- **G** — 비활성 의미: `ARCHIVE_MAX_BYTES = 0`(미설정)이면 용량이 임의로 커도 용량 축출이 일어나지 않는다. `ARCHIVE_RETENTION_DAYS = 0`(미설정)이면 임의로 오래된 아카이브도 나이 축출되지 않는다(명시적 비활성일 때만 무한 적재).
- **H** — 주기성/자기수복: 상한을 초과한 상태로 두고 요청을 보내지 않아도, 1축출 주기 경과 후 용량 불변식이 회복된다(외부 API 호출 없이 주기 실행만으로 수복).

## 검증 스킵 선언 (선택)

- **A · B · C · D · E · F · G · H** — 사유: 전건 mutating — 실제 아카이브 디렉터리를 시딩하고 축출로 삭제하는 비가역 부작용을 수반하므로, 공유 스택이 아니라 **격리 스택 + 제어된 아카이브 시드(크기·시각·`type`)** 에서 판정해야 한다. · 중요도: A·B·C·E·F는 **핵심(load-bearing)**(용량 상한·수동 보존·순서·무해·정합이 이 단위의 3대 보장을 구성), D·G·H는 일반. · 기본 **SKIP** 대상이며 격리 스택이 준비되면 판정한다(INDEX.md의 mutating 격리 게이트 컨벤션과 동류).
