# 비상연락망 서브페이지 (/admin/contacts) 스펙

> 상태: 살아있는 계약 (living spec) · 독자: 스펙 작성자 / 오케스트레이터
> 역할: 관리 IA 재구성의 "비상연락망" 서브페이지 단위. [[admin-management-ia-map]]의
> 한 잎(leaf)이며, 라우팅·내비게이션·접근 경계는 [[admin-routing-navigation-contract]]에
> 의존한다(여기서 재정의하지 않는다).

## 목적 / 의도

관리 영역의 "연락처 관리(비상연락처)" 기능을 독립 서브페이지 `/admin/contacts`로 분리해,
한 화면 = 한 관심사 원칙과 딥링크 가능한 URL을 확보한다. 이 단위는 **순수 IA·표현 계층
변경**이며, 재구성 이전 "연락처 관리" 섹션의 관측 가능한 동작을 **동등 보존**한다.

이 단위는 [[admin-management-ia-map]]의 관통 불변식 3가지를 **참조로 상속**한다:
① 행위 보존, ② 백엔드·API 무변경, ③ admin 전용 접근 경계.

이전 이 섹션은 임의 인증 사용자에게 보였으나, 불변식 ③에 따라 admin 전용으로 고정된다. 행위
동등성 기준선은 **이전 admin 뷰**이며, 비-admin 접근 상실은 의도된 강화다.

## 언어 · 런타임

브라우저 SPA(React). 활성 화면의 단일 진실 소스는 URL(경로)이라는 기존 라우팅 원칙을 따른다.

## 의존 도구 · 시스템

- `/api/contacts` (기존 계약 — [[interface-web-api]] 계약 5). 이 단위는 API 요청/응답
  계약을 바꾸지 않는다.
- 라우팅·내비게이션·접근 경계는 [[admin-routing-navigation-contract]]가 소유한다.

## 입력

- 경로 진입: 허브 항목 탭(드릴다운) 또는 `/admin/contacts` 직접 진입(딥링크)·새로고침.
- 인증 컨텍스트: 현재 사용자 역할(admin / 비-admin / 미인증) — 접합부 계약이 판정.
- 화면 내 사용자 입력: 연락처 추가/수정 폼(이름·전화번호·이메일·이메일 알림 수신 체크),
  수정·삭제 액션.

## 출력 (계약)

- `/admin/contacts` canonical 경로에 마운트되고, admin에게만 렌더된다(접합부 계약).
- 허브(`/admin`)로 돌아갈 수 있다(접합부 계약의 복귀 어포던스/뒤로가기).
- 재구성 이전 "연락처 관리" 섹션과 **행위가 동등한** 비상연락처 관리 화면을 제공한다:
  목록 조회, 추가(검증 포함), 수정, 확인 후 삭제, 이메일 알림 채널 설정과 표시.
- 어떤 서버 엔드포인트·요청/응답 계약도 바뀌지 않는다(각 호출은 동일 admin 인증 헤더를 부착한다).

## 핵심 로직 (동작)

이전 "연락처 관리" 섹션과 동등하게 보장되는 관측 가능한 동작:

- **목록 로드:** 진입 시 등록된 비상연락처 목록을 불러와 표시하고, 로딩 상태가 관측된다.
  목록이 비어 있으면 "등록된 연락처가 없습니다"에 해당하는 빈 상태를 보여주고, 조회가 실패하면
  오류 상태를 표시한다(크래시·빈 화면이 아니다).
- **추가:** 추가 폼에서 이름·전화번호·이메일·"이메일 알림 수신" 체크를 입력해 새 연락처를
  생성한다(`POST /api/contacts {name, phone, email, notifyEmail}`).
- **검증:**
  - 이름은 필수다. 이름이 비면 저장이 거부되고 오류 메시지가 표시된다.
  - 전화번호는 `/^01[016789]-\d{3,4}-\d{4}$/` 형식을 만족해야 저장된다. 형식 불일치 시
    저장이 거부되고 오류 메시지가 표시된다. (이 형식 규칙은 시스템 설정의 현장 정보 전화번호와
    **동일 규칙을 공유**한다 — 공유 유틸 `utils/phone.ts`.)
  - 전화번호 입력은 타이핑 중 하이픈이 자동으로 삽입되는 라이브 포매팅으로 표시된다.
- **수정:** 기존 연락처를 인라인 편집해 이름·전화번호·이메일·이메일 알림 설정을 갱신한다
  (`PUT /api/contacts/{id}`).
- **삭제(확인 경유):** 삭제는 곧바로 지워지지 않고 확인 모달을 거친다. 모달에서 확정하면
  삭제되고(`DELETE /api/contacts/{id}`), 취소하면 아무 변화가 없다.
- **이메일 알림 채널 설정:** "이메일 알림 수신" 체크박스가 이메일 채널 알림 설정값이며,
  목록에서 이 설정이 켜진 연락처에는 "이메일 알림" 배지가 표시된다.
- **이메일 표시:** 이메일이 있는 연락처는 목록에 이메일이 함께 표시된다.

## 검증 단언 (TDD)

관측 가능한 입력→출력으로 판정. `<page>` = `/admin/contacts` 서브페이지.

- **A.** admin이 `/admin/contacts`(딥링크 또는 허브 드릴다운)에 진입하면 `<page>`가 렌더되고
  등록된 비상연락처 목록이 로드된다(빈 목록이면 빈 상태 표시, 조회 실패 시 오류 상태 표시 —
  크래시·빈 화면이 아님). — **핵심**
- **B.** 이름·유효 전화번호를 입력하고 추가하면 새 연락처가 `POST /api/contacts`로 생성되고
  목록에 나타난다. — **핵심**
- **C.** 이름을 비운 채 추가를 시도하면 저장이 거부되고 오류가 표시된다(생성 요청 0건). — **핵심**
- **D.** 전화번호가 `/^01[016789]-\d{3,4}-\d{4}$/`에 맞지 않으면 저장이 거부되고 오류가
  표시된다(생성 요청 0건). — **핵심**
- **E.** 전화번호 입력 시 하이픈이 자동 삽입되는 라이브 포매팅이 관측된다(예: 숫자 나열 입력이
  `010-1234-5678` 형태로 정규화). — 일반
- **F.** 기존 연락처를 수정·저장하면 `PUT /api/contacts/{id}`로 갱신되고 목록에 반영된다. — **핵심**
- **G.** 삭제는 확인 모달을 거친다: 확정 시 `DELETE /api/contacts/{id}`로 삭제되고 목록에서
  사라지며, 취소 시 아무 변화가 없다(삭제 요청 0건). 삭제 요청이 실패하면 오류를 표면화하고
  성공으로 오인하지 않는다(#103 무음 삼킴 방지). — **핵심**
- **H.** "이메일 알림 수신"을 켠 연락처는 목록에 "이메일 알림" 배지가 표시되고, 끈 연락처는
  배지가 없다(이메일 채널 알림 설정의 관측 표현). — **핵심**
- **I.** 이메일이 있는 연락처는 목록에 이메일이 함께 표시된다. — 일반
- **J.** 비-admin·미인증이 `/admin/contacts`에 진입하면 연락처 콘텐츠가 렌더되지 않고 기존
  게이트 동작으로 유도된다([[admin-routing-navigation-contract]] E 스모크 참조; 정본 판정은
  접합부 소유). — **핵심**
