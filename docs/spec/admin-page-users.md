# 사용자 (/admin/users) 스펙

> 상태: 살아있는 계약 (living spec) · 독자: 스펙 작성자 / 오케스트레이터
> 역할: 관리 IA 재구성의 잎(leaf) 단위 하나. 이전 **계정 관리 + 초대 관리** 두 섹션을
> 한 서브페이지로 합친 화면의 계약.

## 목적 / 의도

관리 영역 재구성에서, 이전 세로 스크롤 화면의 **계정 관리**와 **초대 관리** 두 관심사를
성격이 가까운 "사용자" 하나로 묶어 `/admin/users` 서브페이지로 제공한다. 이 단위는
**순수 IA·표현 계층 변경**이며, 위치·내비게이션만 바꾸고 두 섹션의 관측 가능한 동작을
그대로 보존한다.

이 단위는 [[admin-management-ia-map]]의 관통 불변식을 **참조로 상속**한다:
① 행위 보존 ② 백엔드·API 무변경 ③ admin 전용 접근 경계. 도달·복귀·게이트의 방식은
[[admin-routing-navigation-contract]]가 소유하며, 이 스펙은 그 인터페이스에만 의존하고
라우팅·게이트를 재정의하지 않는다.

## 언어 · 런타임

브라우저 SPA(React). URL(경로)이 활성 화면의 단일 진실 소스라는 기존 원칙을 따른다.

## 의존 도구 · 시스템

- 계정 관리 백엔드 계약 (변경 없이 재사용):
  - `GET /auth/pending`, `GET /auth/users`, `POST /auth/approve/{id}`, `POST /auth/reject/{id}`
  - (주의: 이 넷은 `/api/*`가 아니라 `/auth/*` 프리픽스다.)
- 초대 관리 백엔드 계약 (변경 없이 재사용):
  - `GET /api/invitations`, `POST /api/invitations {email}`, `DELETE /api/invitations/{id}`
- 도달·복귀·admin 게이트: [[admin-routing-navigation-contract]].

## 입력

- 경로 진입: `/admin/users` (허브 드릴다운 또는 딥링크). 인증 컨텍스트: admin.
- 계정 조작: 대기 사용자 행의 승인/거절.
- 초대 조작: 이메일 입력·발송, 초대 행의 취소(확인 모달 경유).

## 출력 (계약)

- `/admin/users`에 마운트되고 admin에게만 렌더되며, 허브(`/admin`)로 돌아갈 수 있다
  (접합부 계약 상속).
- 한 페이지 안에서 **계정 관리**와 **초대 관리** 두 관심사를 모두 제공한다.
- 이전 두 섹션과 **행위가 동등**하다 — 아래 핵심 로직이 관측 가능한 계약이다.
- 어떤 서버 엔드포인트·요청/응답 계약도 바뀌지 않는다.

## 핵심 로직 (동작)

### 계정 관리 (accounts)

- 두 하위 구획을 보여준다: **승인 대기** 목록과 **활성 사용자** 목록.
- 승인 대기 목록은 대기 중인 가입 요청을 나열하고, 각 항목에 **승인**·**거절** 어포던스를
  둔다. 승인은 해당 계정을 활성화하고, 거절은 반려한다. 조작 중인 항목은 진행 상태를
  표시하며(행 단위), 조작 대상이 아닌 항목은 영향받지 않는다.
- 활성 사용자 목록은 활성 계정을 나열하고, 역할이 admin인 계정에는 **관리자** 배지를
  표시한다.
- 계정에 대한 삭제 어포던스는 없다.
- 각 목록이 비어 있으면 빈 상태 안내를 표시한다.

### 초대 관리 (invitations)

- 이메일 입력 필드와 발송 어포던스를 둔다. 입력 필드에서 Enter로도 발송할 수 있다.
- 발송 시 이메일에 `@`가 없으면 발송이 거부되고(서버 400 계약과 일관), 인라인 오류를
  표시한다. 유효한 이메일이면 초대가 생성되고 인라인 성공을 표시한다.
- 초대 목록을 나열하고, 각 항목의 상태를 배지로 표시한다:
  `pending`→**대기**, `accepted`→**수락**, `expired`→**만료**, `canceled`→**취소**.
- **취소** 어포던스는 상태가 `pending`인 초대에만 나타난다.
- 취소는 확인 모달을 거쳐 이루어진다 — 확인하면 해당 초대가 취소(삭제)되고, 닫으면
  아무 변화 없이 모달이 사라진다.
- 초대가 없으면 빈 상태 안내를 표시한다.

## 검증 단언 (TDD)

- **A.** `/admin/users`에 admin으로 진입하면 한 화면 안에 **계정 관리**와 **초대 관리**
  영역이 모두 렌더된다. — 핵심
- **B.** 승인 대기 목록의 한 항목을 승인하면 `POST /auth/approve/{id}`가 일어나 해당
  계정이 활성화되고, 거절하면 `POST /auth/reject/{id}`로 반려된다. 조작 중 대상 행은
  진행 상태를 표시하고 다른 행은 영향받지 않는다. — 핵심
- **C.** 활성 사용자 목록에서 역할이 admin인 계정에는 **관리자** 배지가 표시되고,
  비-admin 계정에는 표시되지 않는다. — 핵심
- **D.** 계정 항목에는 삭제 어포던스가 없다. — 일반
- **E.** 이메일에 `@`가 없는 값으로 발송하면 초대가 생성되지 않고(400 계약) 인라인
  오류가 표시된다. `@`를 포함한 유효 이메일이면 `POST /api/invitations`로 초대가
  생성되고 인라인 성공이 표시된다. — 핵심
- **F.** 이메일 입력 필드에서 Enter를 누르면 발송 버튼을 누른 것과 동일하게 발송이
  일어난다. — 일반
- **G.** 초대 목록의 각 상태가 배지로 매핑된다: `pending`→대기, `accepted`→수락,
  `expired`→만료, `canceled`→취소. — 일반
- **H.** **취소** 버튼은 `pending` 상태의 초대에만 나타나고, 다른 상태에는 없다. — 핵심
- **I.** 취소 버튼을 누르면 확인 모달이 뜨고, 확인 시 `DELETE /api/invitations/{id}`
  (204)로 해당 초대가 취소되며, 닫기 시 아무 변화 없이 모달이 사라진다. — 핵심
- **J.** 비-admin·미인증이 `/admin/users`에 진입하면 사용자 관리 콘텐츠가 렌더되지 않고
  기존 게이트 동작으로 유도된다(접합부 계약 E 상속). — 핵심
- **K.** 이 페이지의 어떤 조작도 `/auth/*`·`/api/invitations` 엔드포인트의 요청/응답
  계약을 바꾸지 않는다(기존 계약과 동일 호출). — 핵심
