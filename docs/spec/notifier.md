# notifier 스펙

> 상태: 살아있는 계약 (living spec) · 독자: 스펙 작성자

## 목적 / 의도

- notifier는 산업안전 위기(crisis) 이벤트의 **인명 전파 최종 구간**이다. hw-gateway가 넘겨준 위기 이벤트를 받아, 등록된 모든 비상연락처에게 다채널(KakaoTalk → SMS → 웹 시스템 알람) fallback으로 전달을 시도한다.
- 핵심 의도는 두 가지다:
  1. **유실 방지** — 유효한 위기 이벤트가 접수되면, 각 연락처에 대해 반드시 "외부 채널 성공" 또는 "실패의 로그 기록(+ 시스템 알람 기록 시도)" 중 하나의 관측 가능한 결과가 남아야 한다. 조용히 사라지는 경로(silent fail)는 허용되지 않으며, 모든 실패는 로그에 남는다.
  2. **비차단** — 호출자(hw-gateway)를 절대 기다리게 하지 않는다. 접수 즉시 응답하고 발송은 비동기로 수행한다.
- 부가 의도: 위기 발생 시점의 CCTV 영상을 볼 수 있는 임시 링크를 알림에 포함시키고, 해당 인시던트 시간대의 녹화 세그먼트가 삭제되지 않도록 보호를 요청한다.

## 언어 · 런타임

- Go 단일 바이너리, 표준 라이브러리만 사용 (외부 Go 의존성 없음).
- Docker 컨테이너로 구동, 내부 포트 `:8080`.
- 완전한 stateless — **영구 저장소 없음**. 전송 이력의 유일한 1차 기록은 로그다. 중복 억제(§출력 12)를 위한 최근 incident 캐시는 프로세스 메모리에만 존재하는 휘발성 상태이며, 재시작 시 소멸한다(영속 아님) — 이는 stateless 원칙을 위반하지 않는다. 외부 채널이 모두 실패하면 web-backend의 무인증 internal 라우트(`POST /internal/alarms`)로 시스템 알람을 전송하며, web-backend는 이를 수신해 admin WS `system_alarm` 채널로 브로드캐스트한다(§출력 2). notifier는 전송 시도 + 그 결과(2xx/비 2xx)를 로그로 남긴다.

| 시스템 | 용도 | 계약 소유자 |
|--------|------|-------------|
| web-backend (REST) | 비상연락처 목록 조회, 임시 CCTV 링크 발급, 시스템 알람 기록 시도, site_url 설정 조회, 카메라 목록 조회 | `docs/spec/interface-web-api.md` |
| recording (REST) | 인시던트 녹화 세그먼트 보호 요청 (two-phase 아카이브의 1단계) | `docs/spec/recording.md` (§HTTP API의 `POST /api/archives/protect`) |
| 알림톡(비즈메시지) 채널 (외부) | 1순위 알림 채널. **벤더(발송대행사) 무관 추상 계약** — 전화번호 대상 정보성 메시지 1건을 발송 시도하고 성공/실패를 판정한다. 구체 엔드포인트·인증 헤더·템플릿 변수 규격은 미결(이슈 #26) | 외부 벤더 (미정, 이슈 #26) |
| SMS 채널 (외부) | 2순위 알림 채널. 전화번호 대상 문자 1건 발송 시도·성공/실패 판정 | 외부 벤더 |
| SMTP 서버 (선택) | 독립 이메일 채널 + 범용 이메일 발송 API | 외부 인프라 |

| 변수 | 의미 |
|------|------|
| `WEB_BACKEND_URL` | web-backend 베이스 URL (기본 `http://web-backend:8080`) |
| `FRONTEND_URL` | CCTV 링크 조립 시 site_url 설정이 없을 때의 폴백 베이스 |
| `RECORDING_URL` | 녹화 보호 요청 대상 (기본 `http://recording:8080`, 빈 값이면 보호 요청 생략) |

| `NHN_SMS_APP_KEY`, `NHN_SMS_SECRET_KEY`, `NHN_SMS_SENDER_NO` | NHN SMS 접속 정보. APP_KEY/SECRET_KEY 중 하나라도 빈 값이면 해당 채널은 실패로 처리 |
| `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`, `SMTP_FROM` | 이메일 채널. HOST/FROM 중 하나라도 빈 값이면 이메일 채널 비활성 |
| `DEDUP_WINDOW_SECONDS` | 동일 incident 중복 억제 윈도우(초). 설정 가능하며 기본값은 보수적인 짧은 값(예: 60). `0`이면 중복 억제 비활성 (§출력 12) |
| `CHANNEL_RETRY_MAX` | 외부 채널(알림톡/SMS)의 폴백 전 즉시 재시도 횟수. 설정 가능하며 위기 지연 최소화를 위해 작은 상한(예: 기본 1). `0`이면 재시도 없이 즉시 폴백 (§출력 13) |
| `CHANNEL_RETRY_BACKOFF_MS` | 재시도 간 대기(ms). 설정 가능하며 짧은 값(예: 기본 200). 재시도 총 지연이 채널당 타임아웃 예산(§출력 7)을 침범하지 않도록 작게 유지 |

> 알림톡 관련 변수의 구체 규격(엔드포인트·인증 헤더·템플릿 변수)은 벤더 미선정으로 잠정적이다 — 이슈 #26에서 확정한다. 본 스펙은 이 변수들을 "채널 활성/비활성 + 자격증명 보유 여부"의 설정 표면으로만 다룬다.
> 위 표의 모든 자격증명(API 키·시크릿 키·발신프로필 키·`SMTP_PASS`)은 §출력 9(비밀정보 로그 미노출)의 마스킹 대상이다.

## 입력

### 1. `POST /api/notify` — 위기 이벤트 (주 입력)

- hw-gateway가 MQTT `safety/{siteId}/alert` 페이로드를 그대로 전달한다. **페이로드 필드의 정의·의미는 `docs/spec/interface-mqtt.md`가 소유**하며 이 스펙은 재정의하지 않는다.
- notifier가 강제하는 수용 조건: `siteId`, `deviceId`, `type`, `timestamp` 4개 필드가 모두 비어 있지 않아야 한다. 하나라도 없으면 `400`으로 거절하고 어떤 발송도 하지 않는다.
- `description`이 비면 `"{type} at {siteId}"`, `severity`가 비면 `"unknown"`이 기본값으로 주입된다 — 하위 채널의 템플릿 변수는 절대 빈 값이 되지 않는다.
- `test: true`인 이벤트는 실제 발송 경로를 그대로 타되, 모든 채널의 메시지에 `[테스트]` 표식이 붙는다.

### 2. `POST /api/send-email` — 범용 이메일 발송 (내부 전용)

- Body: `{to, subject, body}` — 셋 다 필수, 누락 시 `400`. 요청 본문 1MB 제한.
- 사설/루프백 IP 대역(Docker 내부망)에서 온 요청만 수용하며, 외부 IP는 `403`으로 거절한다.
- SMTP 미설정 시 `503`.
- `body`는 발송 전 HTML sanitize를 거친다: `<script>`/`<iframe>`은 내용째 제거, `on*` 이벤트 속성 제거, `javascript:` URI 제거, 허용 태그(p, a, br, strong, em, h1~h6, html, head, body) 외의 태그는 껍데기만 제거.

### 3. `GET /healthz` — 헬스체크

- 항상 `200` + `{"status":"ok","service":"notifier"}`.

## 출력 (계약)

1. **즉시 수락 (best-effort 한계 명시)** — 유효한 위기 이벤트에 대해 `/api/notify`는 발송을 기다리지 않고 즉시 `200` + `{"status":"accepted", ...}`를 반환한다. 발송 전 과정은 비동기다. 따라서 `200` 응답과 비동기 dispatch 사이에 프로세스가 크래시·재시작되면 해당 in-flight 이벤트는 유실될 수 있다 — 영속 outbox/큐를 두지 않는 stateless 설계상 **본 스펙은 dispatch를 보장하지 않으며 best-effort 전달만 계약한다**(유실 시 재전송·복구는 계약 대상이 아님). 이는 §목적/의도의 "유실 방지"가 정상 동작 중 silent fail 금지를 뜻하며 크래시 내구성까지 포함하지는 않음을 뜻한다.
2. **연락처별 독립 fallback 체인** — web-backend에서 받아온 각 연락처마다 병렬로, 다음 순서를 보장한다:
   - KakaoTalk 성공 → 해당 연락처 완료.
   - KakaoTalk 실패(일시 오류 포함)·비활성(`KAKAO_ENABLED≠true`)·미설정 → SMS 시도.
   - SMS 성공 → 해당 연락처 완료.
   - SMS까지 실패·비활성·미설정 → web-backend `POST /internal/alarms`로 `notification_failure` 타입 시스템 알람 전송을 **시도**한다. 시도 페이로드에는 연락처 식별 정보, siteId/deviceId, 두 채널의 실패 사유가 포함되며, 호출이 실패하거나 비 2xx 응답이면 그 결과가 로그로 남는다. web-backend는 이 라우트에서 알람을 수신해 admin WS `system_alarm` 채널로 브로드캐스트한다(DB 영속은 없음 — 보장 동작은 "수신 + admin 브로드캐스트"). **본 스펙이 보장하는 것은 "전송 시도 + 결과 로그"까지다.**
   - 즉, 외부 채널이 하나도 설정되지 않은 시스템에서도 각 연락처의 전달 실패는 최소한 notifier 로그로 연락처 수만큼 관측된다.
3. **연락처 간 격리** — 한 연락처의 실패가 다른 연락처의 발송에 영향을 주지 않는다 (연락처별 goroutine 병렬).
4. **이메일은 독립 채널** — `notifyEmail=true`이고 이메일 주소가 있는 연락처에는 KakaoTalk/SMS 체인과 무관하게 병렬로 위기 이메일을 발송한다. 이메일 성패는 fallback 체인 진행에 영향을 주지 않으며, 실패는 로그로만 남는다.
5. **CCTV 임시 링크 degraded 허용** — 알림 발송 전에 web-backend에서 임시 링크(`POST /api/links/temp`)를 1회 발급받아 `{site_url}/view/{token}` 형태로 모든 채널 메시지에 포함한다. 베이스 URL은 web-backend의 `site_url` 설정을 우선하고, 조회 실패·빈 값이면 `FRONTEND_URL`로 폴백한다. **링크 발급이 실패해도 알림 발송은 링크 없이 계속된다** — 링크는 알림의 필요조건이 아니다.
6. **녹화 보호 요청 (알림과 병렬·즉시)** — 녹화 세그먼트 보호는 알림 dispatch와 **병렬로 즉시** 트리거되며, 연락처 fallback 체인의 완료를 기다리지 않는다. 보호 요청은 알림 성패와 독립이고, 보호 요청의 성패 또한 알림 결과에 영향을 주지 않는다(양방향 독립). 인시던트 ID는 `incident_{siteId}_{UTC시각}` 형식이며, 보호 대상은 web-backend에서 조회한 활성(enabled) 카메라 중 **이벤트의 `siteId`에 속한 카메라의 streamKey만**으로 한정한다(다른 site의 세그먼트는 보호 대상에서 제외 — 다중 site 오염 방지). 이벤트 `timestamp`는 RFC3339 또는 `YYYY-MM-DD hh:mm:ss` 형식으로 파싱 가능해야 하며, 파싱 실패·해당 site 카메라 0대·recording 미설정 시 보호 요청만 생략되고 알림 결과에는 영향이 없다.
7. **유한 지연** — 모든 외부 HTTP 호출은 응답 헤더 5초 / 전체 10초 타임아웃을 가진다. 따라서 한 연락처의 체인은 유한 시간 내에 반드시 종결된다.
8. **관측 가능성** — 접수, 채널별 성공/실패, degraded 진입, 시스템 알람 기록 시도의 결과, 보호 요청 결과, dispatch 요약(성공 수/채널별 집계)이 모두 로그로 남는다. 어떤 실패도 로그 없이 소멸하지 않는다. 단 자격증명은 §9에 따라 마스킹되어 기록되며 평문으로는 남지 않는다.
9. **비밀정보 로그 미노출 (보안)** — 외부 채널·이메일 발송에 쓰이는 자격증명(알림톡/SMS의 API 키·시크릿 키·발신프로필 키, `SMTP_PASS`, 그리고 발송 요청에 실리는 `Authorization`/`Bearer` 토큰 등)은 로그·오류 메시지·dispatch 요약 어디에도 **평문으로 남지 않는다**. 진단 목적으로 자격증명 일부를 표기해야 할 때는 마스킹한다(예: 앞 일부만 남기고 나머지를 `***`). 이 보장은 자격증명이 유효하여 발송에 성공하든, 무효/오류로 실패 경로를 타든 **모든 경로에서** 성립한다. (근거: 이슈 #30)
10. **신뢰 경계 — 내부망 격리 (보안)** — notifier의 모든 HTTP 엔드포인트(`/api/notify`, `/api/send-email`, `/healthz`)는 Docker 내부망(`yc-network`) 전용이다. 외부(인터넷)로부터의 차단은 네트워크 격리·게이트웨이의 책임이며, 그것이 대량 주입(위기알림 위조·유료 SMS 남용)에 대한 1차 방어다. 앱은 이 경계를 보완하되 대체하지 않는다:
    - IP 판정은 **소켓 원격 주소(연결 상대)** 기준이며, `X-Forwarded-For` 등 클라이언트가 제어 가능한 헤더는 신뢰하지 않는다. 내부망에 신뢰할 수 있는 리버스 프록시가 없다는 전제이므로, 이런 헤더로 내부 IP 판정을 우회할 수 없다.
    - `/api/send-email`은 그 위에서 사설/루프백 IP 대역(내부망) 요청만 수용하고 외부 IP는 `403`으로 거절한다(입력 §2).
    - `/api/notify`는 별도 앱-레벨 인증(공유 토큰 등)을 요구하지 않는다. 이는 내부망 격리 전제 하의 의도된 계약이며, 통제의 비대칭이 아니라 보안 경계를 네트워크 층에 두기로 한 결정이다.
11. **대상 부재 시에도 silent fail 금지 (최소 1건 시스템 알람)** — 유효한 위기 이벤트를 수락(200)했으나 발송 대상을 확정하지 못하는 경우, 즉 **비상연락처 조회가 실패하거나 결과가 0건**인 경우에도 이벤트가 로그 한 줄로 조용히 소멸해서는 안 된다. 이때 notifier는 web-backend `POST /internal/alarms`로 대상 없음/조회 실패를 알리는 시스템 알람(`target_unavailable` 류 타입, siteId/deviceId와 사유 포함) 전송을 **최소 1건 시도**하고 그 결과(2xx/비 2xx/호출 실패)를 로그로 남긴다. 즉 "연락처가 없어서 아무도 이 위기를 모르는" 상태 자체가 관측 가능한 알람으로 승격된다. (인시던트 원본 기록은 hw-gateway 별도 경로 소관이며 본 항목은 그것을 대체하지 않는다.)
12. **동일 incident 중복 억제 (dedup/throttle)** — 동일 incident 키 `(siteId, deviceId, type)`가 `DEDUP_WINDOW_SECONDS`(기본 60s, 설정 가능) 이내에 재수신되면, notifier는 중복으로 판정하여 그 재수신 이벤트에 대한 연락처 발송(알림톡/SMS/이메일/시스템 알람 dispatch)을 **억제**한다. 억제된 이벤트도 접수 자체는 `200`으로 응답되며, 억제 사실(중복 판정 + 원 incident 키)이 로그로 남는다. 윈도우는 위기 알림을 과도하게 삼키지 않도록 보수적으로 짧게 유지하며, `DEDUP_WINDOW_SECONDS=0`이면 억제를 하지 않는다. 억제 상태는 휘발성 in-memory(§런타임)이므로 재시작 후에는 중복 판정이 초기화된다(best-effort). **동일 키의 최초 이벤트는 절대 억제되지 않는다.**
13. **외부 채널 즉시 재시도 (폴백 전)** — 외부 채널(알림톡·SMS)의 발송 시도가 일시적 오류(네트워크 오류·타임아웃·5xx 류)로 실패하면, 다음 폴백 단계로 강등하기 전에 같은 채널로 **즉시 재시도**한다. 재시도 횟수는 `CHANNEL_RETRY_MAX`(기본 1, 설정 가능), 재시도 간 대기는 `CHANNEL_RETRY_BACKOFF_MS`(기본 200ms, 설정 가능)이다. 위기 지연 최소화를 위해 재시도 상한과 대기는 작게 유지하며, 재시도를 포함한 채널 전체 소요가 §7의 타임아웃 예산 안에서 유한하게 종결된다. `CHANNEL_RETRY_MAX=0`이면 재시도 없이 곧바로 폴백한다. 재시도 발생 및 최종 성패는 로그로 남는다. (명백한 영구 실패 — 예: 채널 비활성·자격증명 미설정 — 는 재시도 대상이 아니며 즉시 폴백한다.)

## 핵심 로직 (동작)

1. 위기 이벤트 수신 → 필수 4필드 검증(실패 시 400 종료) → 선택 필드 기본값 주입 → 중복 억제 판정(§출력 12: 윈도우 내 동일 `(siteId,deviceId,type)`면 억제) → 즉시 200 응답, 이후 비동기 진행.
2. web-backend에서 비상연락처 목록 조회. 조회 실패 또는 0건이면 연락처 발송은 없으나, silent fail 금지를 위해 시스템 알람(`target_unavailable` 류) 전송을 최소 1건 시도하고 그 결과를 로그로 남긴다(§출력 11).
3. 녹화 세그먼트 보호 요청을 알림 dispatch와 병렬로 즉시 트리거한다(§출력 6, best-effort). 대상은 이벤트 `siteId`에 속한 활성 카메라 streamKey로 한정.
4. 임시 CCTV 링크 1회 발급 시도 → 성공 시 링크 조립, 실패 시 링크 없이 진행 (degraded).
5. 연락처마다 병렬 실행:
   - 이메일 대상이면 이메일을 별도 병렬 발송.
   - KakaoTalk(활성 시) → SMS(활성 시) → 시스템 알람 기록 시도 순서의 순차 fallback. 각 외부 채널은 일시 오류 시 §출력 13에 따라 즉시 재시도 후 폴백한다. 성공한 첫 채널에서 종료.
6. 전 연락처 완료 대기 후 dispatch 요약 로그.

## 검증 단언 (TDD)

전제: sentinel 스택이 기동 중이고, 단언은 Docker 내부망의 다른 컨테이너에서 notifier(`http://sentinel-notifier:8080` 또는 compose 서비스명 `notifier:8080`)로 직접 주입한다. 관측은 `docker compose logs notifier`로 한다.

- A. **즉시 수락**: 필수 4필드를 갖춘 위기 이벤트를 POST하면 1초 이내에 `200`과 body `"status":"accepted"`가 반환된다.
  ```bash
  curl -s -o /dev/null -w '%{http_code} %{time_total}\n' -X POST http://notifier:8080/api/notify \
    -H 'Content-Type: application/json' \
    -d '{"siteId":"site1","deviceId":"TEST-01","type":"gas_leak","timestamp":"2026-07-02T00:00:00Z","test":true}'
  ```
  → OK: `200`이고 time_total < 1.0. NOK: 그 외.
- B. **불량 입력 거절**: `siteId`(또는 `deviceId`/`type`/`timestamp`) 누락 페이로드 POST → `400` 반환, notifier 로그에 어떤 채널 발송 시도도 나타나지 않는다.
- C. **최후 보루 시도 + 결과 로그**: `KAKAO_ENABLED`/`SMS_ENABLED`가 둘 다 true가 아닌 상태에서 연락처가 N(≥1)명일 때 유효 이벤트 1건 주입 → notifier 로그에 시스템 알람 전송 시도의 결과 기록(성공 또는 `System alarm failed`)이 정확히 N건 남는다 (연락처당 1건). OK/NOK: 알람 시도 결과 로그 건수 = 연락처 수. (web-backend `POST /internal/alarms` 수신 라우트가 무인증 2xx로 응답하므로 각 시도는 성공 로그로 종결되고, 접속 중 admin WS 클라이언트는 `system_alarm` 메시지를 수신한다. 알람의 DB 적재 여부는 단언하지 않는다.)
- D. **링크 degraded**: 임시 링크 발급이 실패하는 상황(web-backend 링크 API 불가)에서 유효 이벤트 주입 → notifier 로그에 degraded 진입이 기록되고, 그럼에도 각 연락처에 대한 채널 시도(또는 시스템 알람)가 관측된다. NOK: 링크 실패로 발송 자체가 중단되는 경우.
- E. **테스트 표식**: `test:true` 이벤트 주입 → 발송되는 SMS/이메일 본문(또는 KakaoTalk 변수)의 설명에 `[테스트]` 표식이 포함된다.
- F. **기본값 주입**: `description`/`severity` 없이 유효 이벤트 주입 → notifier 로그에 fallback 주입이 기록되고, 발송 메시지의 해당 자리들이 빈 문자열이 아니다.
- G. **이메일 API 접근 통제**: 외부(공인) IP에서 `/api/send-email` POST → `403`. 내부망에서 `<script>alert(1)</script>` 포함 body로 POST → 발송된 메일 본문에 `<script>`가 존재하지 않는다.
- G2. **위조 헤더로 우회 불가 (신뢰 경계)**: 외부(공인) IP에서 `/api/send-email`에 `X-Forwarded-For: 127.0.0.1`(또는 `10.0.0.1` 등 사설/루프백 IP)를 위조해 담아 POST해도 여전히 `403`으로 거절된다. 즉 접근 판정은 소켓 원격 주소만을 근거로 하며, 클라이언트가 제어 가능한 프록시 헤더로는 내부 판정을 통과시킬 수 없다.
  ```bash
  # 외부 IP에서 실행 — 위조 헤더를 담아도 403이어야 한다
  curl -s -o /dev/null -w '%{http_code}\n' -X POST http://<notifier-외부주소>/api/send-email \
    -H 'Content-Type: application/json' -H 'X-Forwarded-For: 127.0.0.1' \
    -d '{"to":"a@b.c","subject":"x","body":"y"}'
  ```
  → OK: `403`. NOK: `403` 이외(위조 헤더로 우회됨).
- H. **헬스체크**: `curl -s http://notifier:8080/healthz` → `200` + `"status":"ok"`.
- I. **녹화 보호 트리거 (site 한정)**: 이벤트 `siteId`에 속한 활성 카메라가 1대 이상일 때 유효 이벤트 주입 → notifier 로그에 인시던트 ID(`incident_{siteId}_...`)와 보호 대상 카메라 수를 포함한 보호 요청 기록이 남는다. 서로 다른 두 site A/B의 카메라가 존재할 때 site A 이벤트를 주입하면 보호 대상 streamKey는 site A 카메라만 포함하고 site B 카메라는 포함하지 않는다. recording이 내려가 있어도 알림 결과(단언 C 등)는 변하지 않는다.
- J. **연락처 없음 시 종결**: 연락처 0건 상태에서 유효 이벤트 주입 → `200` accepted는 반환되고, notifier 로그에 "대상 없음으로 스킵" 기록이 남으며 어떤 채널 호출도 발생하지 않는다.
- K. **비밀정보 마스킹 (핵심)**: 알림톡/SMS/SMTP 중 하나 이상의 자격증명이 실제로 설정된 상태에서 유효 이벤트를 1회 이상 주입해 발송 경로(성공·실패 무관)를 태운 뒤, `docker compose logs notifier` 전체에서 설정된 자격증명 평문 값을 grep하면 매칭이 **0건**이어야 한다.
  ```bash
  # 설정된 자격증명이 환경변수 KAKAO_API_KEY / NHN_SMS_SECRET_KEY / SMTP_PASS 에 있을 때
  docker compose logs notifier | grep -F -e "$KAKAO_API_KEY" -e "$NHN_SMS_SECRET_KEY" -e "$SMTP_PASS"
  ```
  → OK: 매칭 0건 (자격증명 원문 미노출). NOK: 어느 한 자격증명 원문이라도 로그에 출현.
- L. **비활성 채널 스킵**: `KAKAO_ENABLED≠"true"`(또는 `SMS_ENABLED≠"true"`)인 상태에서 유효 이벤트 주입 → notifier 로그에 해당 채널의 발송 시도 기록이 나타나지 않고, 곧바로 다음 폴백 단계(SMS 또는 시스템 알람)로 진행한 기록이 남는다. NOK: 비활성 채널에 대한 외부 발송 시도가 로그에 관측되는 경우.
- M. **대상 부재 시 시스템 알람 (silent fail 금지)**: 비상연락처가 0건(또는 연락처 조회가 실패)인 상태에서 유효 이벤트 1건 주입 → `200` accepted가 반환되고, notifier 로그에 대상 부재/조회 실패에 대한 시스템 알람(`target_unavailable` 류) 전송 시도의 결과 기록이 **최소 1건** 남는다. OK: 대상 부재임에도 시스템 알람 시도 결과 로그가 존재. NOK: 로그 한 줄만 남고 어떤 시스템 알람 전송 시도도 관측되지 않음.
- N. **중복 억제 (dedup)**: `DEDUP_WINDOW_SECONDS`가 양수(예: 60)인 상태에서 동일 `(siteId,deviceId,type)` 이벤트를 윈도우 내에 연속 2회 주입 → 두 요청 모두 `200`이지만, 실제 연락처 발송(채널 시도 또는 시스템 알람 dispatch)은 **1회분만** 관측되고 두 번째 이벤트에 대해서는 중복 억제 로그가 남는다. 이어서 `DEDUP_WINDOW_SECONDS=0`으로 재구성하면 동일 이벤트 2회 주입 시 발송이 2회분 관측된다. OK: 윈도우 활성 시 최초 1회만 발송·2번째 억제. NOK: 윈도우 내인데도 매 요청마다 발송이 반복되거나, 최초 이벤트가 억제되는 경우.
- O. **채널 재시도 후 폴백**: 알림톡 채널이 일시 오류(5xx/타임아웃)를 반환하도록 구성하고 `CHANNEL_RETRY_MAX≥1`인 상태에서 유효 이벤트 주입 → notifier 로그에 알림톡 발송이 재시도된 기록(같은 채널 2회 이상 시도)이 남은 뒤 SMS(또는 다음 폴백)로 진행한다. `CHANNEL_RETRY_MAX=0`으로 재구성하면 재시도 없이 즉시 폴백한 기록이 남는다. 어느 경우든 한 연락처의 체인은 §7 타임아웃 예산 내에서 유한 시간에 종결된다. OK: 재시도 설정이 로그에 반영. NOK: 일시 오류인데 재시도 없이 즉시 강등(설정≥1인데도)되거나, 재시도가 타임아웃 예산을 초과해 체인이 종결되지 않는 경우.
- P. **보호 요청 병렬성 (알림 완료를 대기하지 않음)**: 외부 채널이 느리게 응답(발송이 여러 초 지연)하도록 구성한 상태에서 유효 이벤트 주입 → notifier 로그상 녹화 보호 요청 기록이 전 연락처 dispatch 요약 로그보다 **먼저(또는 그와 독립적으로 즉시)** 나타난다. OK: 보호 요청이 dispatch 완료 이후로 밀리지 않음(병렬·즉시). NOK: 보호 요청이 모든 연락처 발송이 끝난 뒤에야 나타나는 경우.

## 검증 스킵 선언

- **알림톡 벤더 배관 정합** — 사유: 발송대행사(벤더) 미선정으로 알림톡의 구체 엔드포인트·인증 헤더·템플릿 변수 규격이 미확정. 알림톡 채널은 **추상 계약**(전화번호 대상 정보성 메시지 1건 발송 시도 → 성공/실패 판정 → 폴백)으로만 검증하며, 특정 벤더의 요청 스키마 정합(경로·헤더·응답 파싱)은 지금 검증하지 않는다. · 중요도: 일반 · 해소 조건: 이슈 #26 — 발송대행사 선정 후 벤더 스펙을 확정하면 구체 계약과 단언을 추가한다.

## ⚠️ 리뷰 필요 (의도 불확실)

1. **채널 활성 플래그의 기본값이 '꺼짐'** — KakaoTalk/SMS는 `KAKAO_ENABLED`/`SMS_ENABLED`가 문자열 `"true"`일 때만 시도된다. 서비스 가이드(`docs/services/notifier.md`)의 환경변수 표에는 이 두 변수가 없고 "API 키가 빈 값이면 스킵"으로만 기술되어 있다. 운영 배포에서 이 플래그를 설정하지 않으면 API 키가 있어도 두 외부 채널이 전부 비활성화되어 모든 위기 알림이 시스템 알람으로만 남는다. 의도된 안전장치(비용 방지)인지, 문서 누락인지 확인 필요.
2. **응답의 `contactCount`가 항상 0** — `/api/notify` 200 응답 body에 `contactCount: 0`이 고정으로 들어간다 (비동기라 시점상 알 수 없음). 의미 없는 필드를 계약에 남길지, 제거할지 결정 필요.
