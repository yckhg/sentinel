# H/W Spec — Sentinel PC (CCTV 4대 기준)

작업장 1곳 = **PC 1대** = Sentinel 인스턴스 1개 (온프레미스, 폐쇄망).
폼팩터는 자유 — 발열/공간 이점 때문에 **미니 PC가 선호**되나 일반 데스크탑도 무방.
본 문서는 **CCTV 4대** 기준 단일 호스트 사양을 정의한다.

## 설계 결정 (2026-07-01)

**기본은 무변환(`-c copy`), 변환은 폴백(fallback)으로만 쓴다.**
- **최선 = 무변환.** 카메라를 규격(H.264 Baseline + B-frame off + AAC)으로 설정해 **전부 copy 처리** → CPU ~0, 화질 손실 0, 지연 최소, PC 저사양.
- 프로 카메라(Hikvision/한화/Dahua)는 웹UI에서 이 설정 가능 → 무변환 달성 가능.
- **폴백 = 변환.** 규격을 못 맞추는 카메라(설정 불가·H.265 전용)만 자동으로 재인코딩 → 그 카메라만 ~1코어.
- 무변환 경로(`-c copy`)는 **현재 코드에 이미 구현됨** — 코드 변경 불필요. 자동 폴백만 미구현(아래 참조).

## 사양을 좌우하는 전제

| 항목 | 사실 | 부하 |
|------|------|------|
| CCTV 수신 | cctv-adapter가 RTSP→RTMP. **호환 카메라=copy(CPU~0)**, 비호환만 변환 폴백 | 변환 필요분만 ~1코어/대 |
| 영상 스트리밍 | RTMP→HLS **remux only, 무트랜스코딩** (nginx-rtmp) | CPU 최소 |
| 녹화 | FFmpeg `-c copy` (streaming RTMP 복사, 재인코딩 없음) | CPU 낮음 |
| 위기 감지 | **센서 하드웨어/엣지**에서 판단 → MQTT 전송. PC에서 영상 AI 추론 **없음** | **GPU 불필요** |
| DB | SQLite 단일 파일 (메타데이터 위주) | 무시 가능 |

→ GPU 불필요. 전 카메라 무변환(copy)이면 매우 가볍고, **변환이 필요한 카메라 수만큼만** CPU를 더 쓴다.

## 카메라 호환성

- **최선(권장): 무변환.** 카메라를 H.264 + Smart Codec/H.264+/H.265+ **끄기**(→ 브라우저 재생 가능한 H.264 확보) + 오디오 **AAC**로 설정하면 `-c copy` 직결 → CPU~0, 화질·지연 최상.
  - 허브는 remux-only라 코덱을 검사하지 않고 적법한 H.264(B-frame 포함)를 통과·서빙한다 — **B-frame이 있어도 끊기지 않는다.** 무변환을 권장하는 이유는 허브 호환이 아니라 **CPU 절약(트랜스코딩 회피)**이다. 저지연이 특히 중요하면 Baseline(B-frame 없음)이 재정렬 지연을 줄여 유리하다(선택).
  - 프로 카메라(Hikvision DS-2CD2143G2-I, 한화 QNO-8020R, Dahua)는 웹UI/ONVIF에서 설정 가능.
- **폴백(안전망): 변환.** 위 설정이 불가능하거나 H.265 전용인 카메라만 자동 변환 → 해당 카메라 ~1코어.
- **결론: 호환 프로 카메라로 표준화 → 무변환이 최선.** 변환 폴백은 예외 카메라·오설정 대비 보험.

## 권장 PC 사양

CPU는 **변환이 필요한 카메라 수**로 결정된다.
- **전 카메라 무변환(copy): N100/4코어로도 8대 OK** (회장님 방침 = 이 경로).
- 변환 폴백을 실제로 돌릴 여유까지 감안 시 아래 표 (변환 부하: 4MP ~1코어/대, 1080p ~0.5코어/대).

| 구분 | 최소 (4대) | **권장 (4대+8대 여유)** | 최대 |
|------|-----------|-------------------------|------|
| CPU | **6코어 실코어** (Ryzen 5 / i5) | **8코어 16스레드 (Ryzen 7 / i7급)** | 8코어+ |
| RAM | 8 GB | **16 GB** | 32 GB |
| OS 디스크(SSD) | 128 GB NVMe | **256 GB NVMe** | 512 GB |
| 녹화 디스크 | 1 TB (2.5" SATA/2nd M.2) | **1~2 TB** | 4 TB+ |
| GPU | 불필요 | **불필요** | — |
| 네트워크 | 기가비트 LAN | **기가비트 LAN** | — |

- CPU 예산: 4대 ≈ 3~4코어 + OS/서비스 1~2코어 → **6코어면 4대 충분**. 8대 확장까지 여유 두려면 **8코어**.
- **N100/N150(4C/4T)은 변환 8대를 감당 못 함** → "변환 없는 최소 구성"이나 소수 카메라 저장 중심에만.
- 해상도가 변환 부하를 좌우 → 8대까지 갈 계획이면 **2~4MP 권장**(5MP↑는 코어 더 필요).
- 녹화 디스크는 값싼 2.5" SATA HDD 장착 가능한 폼팩터가 유리(아래 제품 참조).

## 제품 추천 (국내 조달 기준, 2025~2026)

가격은 대략치. **1~2대 선구매 → 파이프라인 실측 통과 후 4~8대 확장** 권장.

### 미니 PC (공식 수입/AS 우선 + 2.5" 베이)

**방침 2개 확정:**
1. **무변환 우선** → 6코어면 충분(무변환만이면 4코어도 OK). 8코어 몬스터 불필요.
2. **공식 수입처/AS 우선** → 미니PC 특화(Beelink·MINISFORUM)는 **진짜 정식 수입처가 아님**(MINISFORUM=홍콩 직영 송환 A/S·직구/쿠팡분 보증 제외, Beelink=병행). 24/7 안전장비엔 **엔터프라이즈 마이크로(Dell·HP·Lenovo 코리아 직판·전국 서비스망)** 가 정석.

### IP 카메라 (변환 폴백 채택 → 코덱 호환성 무관)

변환을 PC가 하므로 **표준 RTSP만 나오면 무엇이든 OK** (H.265·B-frame 상관없음).
선택 기준 = 화질/PoE/국내 AS/가성비. 단 **고해상도일수록 변환 CPU↑** → 8대 계획이면 2~4MP 권장.

| # | 브랜드 | 모델 코드 | 해상도 | PoE | 대략가(KRW) | 비고 |
|---|--------|-----------|--------|-----|-------------|------|
| 1 | Hikvision | **DS-2CD2143G2-I** | 4MP | O | 10~12.5만 | 가성비 표준 |
| 2 | 한화비전 Wisenet | **QNO-8020R** | 5MP | O | 약 23만 | **국산 AS 최상** |
| 3 | Dahua | **IPC-HFW2441S-S** (단종품 HFW2431 후속) | 4MP | O | 6~9만 | 최저가 |
| 4 | TP-Link | **VIGI C340** | 4MP | O | 7~10만 | 가성비 |
| 5 | Reolink | **RLC-520A** | 5MP | O | 9~11만 | 변환 채택으로 이제 문제없음 |

- **가성비 표준: ① Hikvision DS-2CD2143G2-I**, **국산 AS 우선: ② 한화 QNO-8020R.**
- Reolink 등도 **변환 폴백(비-H.264/H.265 소스 정규화) 채택으로 5개 모델 모두 사용 가능.** (허브가 B-frame을 거부하는 문제는 없으며, 폴백이 필요한 경우는 HEVC 등 브라우저 비재생 코덱 정규화다.)

**Hikvision 정식수입/AS 주의 (중요):**
- 국내 공식 법인 **하이크비전코리아(판교)** + 공식 서비스센터 존재. 총판: **씨넥스존(공식몰 hikvisionmall.co.kr)·준영테크·SK네트웍스서비스**.
- **오픈마켓 최저가(11.5만~)는 병행수입일 수 있어 국내 공식 A/S 불가.** 정식 A/S를 원하면 **총판 채널에서 "정식수입" 여부를 판매자에게 명시 확인** 후 구매(리스팅에 정식/병행 표기 없음).
- **중국산 CCTV 규제**: 민간 산업안전용은 **법적 문제 없음**. 단 국정원·한전·국방은 공공/핵심인프라에서 중국산 교체 권고·진행 중 → **발주처가 미국계·공공·방산 연계 가능성이 있으면** 한화 Wisenet(NDAA/TAA 준수, 비중국) 검토. 단 **한화가 동급 4MP 기준 약 3.5배** 가격.

## 코드 변경 필요 (선행 작업)

무변환(`-c copy`) 경로는 **이미 구현됨** — 호환 카메라만 쓰면 코드 변경 불필요.
자동 폴백(비호환 카메라 감지 → 변환)만 **미구현**. `services/cctv-adapter/main.go`가 현재 `-c copy` 고정.
- 권장 구현: 스트림 시작 시 ffprobe로 코덱 판별 → 브라우저 재생 가능한 H.264면 `-c copy`, 비-H.264(HEVC 등)면 `-c:v libx264 -tune zerolatency -bf 0 -preset veryfast -c:a aac`로 그 카메라만 정규화 전환. (코덱 정규화 = 어댑터 책임 — 허브는 트랜스코딩하지 않음.)
- 문서엔 방법만 기술됨(`docs/services/cctv-adapter.md`).
- 폴백 없는 현 상태에선 카메라가 **H.264 + AAC**여야 동작한다(B-frame 유무는 무관 — 허브가 수용). 비-H.264(HEVC) 카메라는 폴백 구현 전까지 브라우저 재생 불가.

## 녹화 저장 용량 (핵심 변수)

CPU/RAM보다 **디스크 산정이 핵심**. 카메라 비트레이트 × 녹화 정책에 좌우됨.
(문서에 비트레이트 명시 없음 — 통상 산업용 CCTV 기준 추정치)

### A. 60분 롤링 (현재 기본값 `ROLLING_WINDOW_MINUTES=60`)
- 최근 60분만 유지 후 자동 삭제 → **상시 약 6 GB만 점유** (4대 합, 2Mbps 기준)
- 사건 아카이브(MP4)만 영구 보관 → 용량 = 사고 빈도 × 보관 정책

### B. 24시간 상시 녹화 (정책 변경 시)
| 비트레이트 | 카메라당/일 | 4대/일 | 30일 보관 |
|-----------|------------|--------|-----------|
| 2 Mbps (1080p) | 21.6 GB | 86 GB | **약 2.6 TB** |
| 4 Mbps (고화질) | 43.2 GB | 172 GB | **약 5.2 TB** |

## 확정 필요 항목

정확한 디스크 용량 확정을 위해 필요:
1. **카메라 해상도/비트레이트** (1080p 2Mbps vs 4K/고비트레이트)
2. **녹화 정책** (60분 롤링 유지 vs 상시 N일 보관)

## 네트워크

- 카메라 4대 지속 트래픽 ≈ 8 Mbps (2Mbps × 4) → 기가비트 LAN 여유 충분
- 내부 LAN only, WAN 불필요 (KakaoTalk/SMS 알림 시에만 외부 경로)
