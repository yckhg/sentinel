## 2026-03-22 18:54 KST - US-001 (RESOLVED)
- **Root cause**: nginx-rtmp module v1.2.2 drops RTMP connections when H.264 B-frames are present in the FLV container. The composition time offsets (CTS) from B-frames cause the module to reset the connection after ~5 seconds. This was NOT an audio issue — the error appeared on audio packets simply because audio happened to be the packet being written when the connection was dropped.
- **Key diagnostic steps**:
  - Tested video-only (no audio): cam-2 (no B-frame issues at 60kbps) worked, cam-1 (127kbps, 29.97fps) still failed at ~5s → proved audio was not the cause
  - Tested cam-1 with libx264 re-encode (bframes=0, -tune zerolatency): streamed stably for 15+ seconds → confirmed B-frames were the root cause
- **Fix**: Changed FFmpeg from `-c copy` to `-c:v libx264 -preset ultrafast -tune zerolatency -b:v 300k -g 60 -c:a aac -b:a 48k`
- **Files changed**: services/youtube-adapter/main.go, services/streaming/nginx.conf
- **Learnings for future iterations:**
  - nginx-rtmp v1.2.2 (Alpine apk) cannot handle H.264 B-frames in FLV — always use -tune zerolatency or -bf 0
  - "Connection reset by peer" on audio muxing is misleading — the actual cause can be video-related
  - The streaming Dockerfile uses `alpine:3.19 + apk add nginx nginx-mod-rtmp`, NOT a pre-built tiangolo image
  - Exit status 152 from FFmpeg = signal 24 (SIGXCPU) — appears to be how FFmpeg exits on RTMP connection reset
  - go vet must be run via Docker: `docker run --rm -v "$PWD":/app -w /app golang:1.22-alpine go vet ./...`
